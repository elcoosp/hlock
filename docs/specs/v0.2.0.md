# HLOCK Specification v2.0

## 1. Overview
HLOCK v2.0 introduces critical robustness and developer experience (DX) improvements over v1.0. While maintaining the core hybrid line-oriented philosophy, v2.0 adds payload versioning, dynamic hash lengths, and payload integrity checks to protect against file corruption and future format drift. 

**Breaking Change:** HLOCK v2.0 parsers will intentionally reject v1.0 payloads due to the introduction of the version header.

## 2. File Structure
Unchanged from v1.0. The file is UTF-8 encoded, LF line endings (`\n`). One package per line.

```text
<package_name>\t<payload_base64url>
```

## 3. The Binary Payload
The binary payload (before Base64URL encoding) has been expanded to support schema evolution and corruption detection. All multi-byte integers remain Unsigned LEB128 (Varint).

### 3.1 Payload Layout

| Field | Type | Size | Description |
| :--- | :--- | :--- | :--- |
| `Version` | Byte | 1 byte | Payload schema version. **MUST be `0x01`** for v2.0. Allows future parsers to safely reject unsupported formats. |
| `Major` | Varint | 1-9 bytes | Package major version. |
| `Minor` | Varint | 1-9 bytes | Package minor version. |
| `Patch` | Varint | 1-9 bytes | Package patch version. |
| `HashLen` | Byte | 1 byte | Length of the integrity hash in bytes (e.g., `32` for SHA-256, `16` for truncated SHA-256). |
| `Hash` | Bytes | `HashLen` bytes | The raw integrity hash bytes. |
| `DepCount`| Varint | 1+ bytes | Number of direct dependencies. |
| `DepIndices`| `DepCount` x Varint | `DepCount` bytes+| 0-based line indices of dependencies in the sorted lockfile. |
| `CRC32` | Bytes | 4 bytes | Standard CRC32 (IEEE 802.3) checksum of **all preceding bytes** in the payload (from `Version` through the last `DepIndex`). Stored in little-endian byte order. |

### 3.2 Integrity Check (CRC32)
To ensure the Base64URL string hasn't been truncated or corrupted (e.g., via a bad `git merge` or disk fault), a CRC32 is appended.
1. The parser reads the Base64URL string and decodes it to bytes.
2. It separates the last 4 bytes (the CRC32).
3. It calculates the CRC32 of the remaining bytes.
4. If the calculated CRC32 does not match the trailing 4 bytes, the parser MUST throw an `IntegrityCheckFailed` error for that specific line, allowing the rest of the file to potentially be parsed or reported.

## 4. Dependency Index Resolution
Unchanged from v1.0. Packages are sorted alphabetically by `<package_name>` and assigned an implicit 0-based line index. Dependencies reference this index.

## 5. Public API Specification

### 5.1 Core Data Structures

```rust
pub struct Package {
    pub name: String,
    pub major: u64,
    pub minor: u64,
    pub patch: u64,
    pub hash: Vec<u8>, // Changed from [u8; 16] to support dynamic lengths
    pub dependencies: Vec<String>,
}
```

### 5.2 Error Handling

Errors must be rich, context-aware, and implement `std::error::Error`.

```rust
pub enum Error {
    /// Failed to read or write the underlying file
    Io(std::io::Error),
    
    /// A line in the lockfile is missing the required '\t' delimiter
    MissingDelimiter { line_number: usize },
    
    /// The Base64URL payload on a specific line is syntactically invalid
    InvalidBase64 { line_number: usize },
    
    /// The payload schema version is not supported by this parser
    UnknownPayloadVersion { line_number: usize, version: u8 },
    
    /// The calculated CRC32 does not match the trailing 4 bytes
    IntegrityCheckFailed { line_number: usize },
    
    /// A dependency index points outside the bounds of the parsed lockfile
    InvalidDependencyIndex { package: String, index: u64 },
    
    /// A package name was referenced in dependencies but never defined
    MissingPackage { package: String, missing_dep: String },
}
```

### 5.3 Core Functions (I/O Decoupled)

The primary API must operate on strings, separating serialization logic from file system operations.

*   `fn serialize(packages: &mut Vec<Package>) -> Result<String, Error>`
    *   Sorts packages alphabetically.
    *   Builds the index map.
    *   Iterates packages, calling internal `pack_payload` and `base64url::encode`.
    *   Joins lines with `\n`.
*   `fn deserialize(lockfile_content: &str) -> Result<Vec<Package>, Error>`
    *   Iterates lines with `lines().enumerate()`.
    *   Splits on `\t`.
    *   Decodes Base64URL -> Validates CRC32 -> Unpacks payload.
    *   Builds an index map during iteration to resolve dependency indices back to strings.
    *   Returns the fully resolved `Vec<Package>`.

### 5.4 Convenience I/O Functions

Thin wrappers around the Core Functions:

*   `fn write_lockfile(path: &Path, packages: &mut Vec<Package>) -> Result<(), Error>`
    *   Calls `serialize()`.
    *   Writes the resulting string to disk using `std::fs::write`.
*   `fn read_lockfile(path: &Path) -> Result<Vec<Package>, Error>`
    *   Reads file to string using `std::fs::read_to_string`.
    *   Calls `deserialize()`.
