# HLOCK Specification v0.5.0

## 1. Overview
HLOCK v0.5.0 introduces the **Merge-Safe Content-Addressable Graph**, **First-Class Feature Flags**, and **Platform-Targeted Dependencies**. 

This is the largest paradigm shift since the project's inception. By replacing line-based indexing with Content-Addressable IDs (based on FNV-1a 64-bit hashes of `name@version`), HLOCK becomes natively compatible with `git merge`. Two developers can add completely different dependency trees on separate branches, and Git will merge the lockfile automatically without binary corruption or textual conflicts.

Furthermore, v0.5.0 introduces a local feature string table per package and platform target constraints, allowing lockfiles to perfectly describe complex matrices like "resolve `tokio` with `full` and `macros` features, but only fetch the `x86_64` binary for `esbuild`".

**Breaking Change:** HLOCK v0.5.0 parsers will intentionally reject v0.4.0 and older payloads.

## 2. File Structure
Unchanged conceptually, with the addition of the `@feature` directive.

```text
<header directives>
<empty line>
<package lines>
```

### 2.1 The Metadata Header
*   `@source <idx> <uri>`: Unchanged.
*   `@override <name> <ver> -> <ver>`: Unchanged.
*   `@feature <name> [flag1,flag2,...]` (NEW): Defines the top-level feature resolution for a specific package. These are the features activated by the root workspace. 

**Example Header:**
```text
@source 0 https://registry.npmjs.org/
@source 1 workspace
@feature serde derive,rc
@feature tokio full
```

## 3. Content IDs (Merge Safety)
To achieve Git-merge safety, dependencies are no longer referenced by their 0-based line index in the file. They are referenced by a deterministic **Content ID**.

*   **Algorithm:** FNV-1a 64-bit.
*   **Input String:** `<package_name>@<major>.<minor>.<patch>` (e.g., `lodash@4.17.21`).
*   **Storage:** The resulting 64-bit integer is stored as a little-endian 8-byte array in the binary payload.
*   **Resolution:** During serialization, the parser computes the Content ID for every package and maps `ContentID -> PackageName`. During deserialization, the parser builds the inverse map. If a dependency references a Content ID that does not exist in the file, it is a `MissingContentId` error.

## 4. The Binary Payload (v0.5 Schema)

### 4.1 Payload Layout

| Field | Type | Size | Description |
| :--- | :--- | :--- | :--- |
| `Version` | Byte | 1 byte | Payload schema version. **MUST be `0x04`** for v0.5.0. |
| `SourceIdx`| Varint | 1+ bytes | Index mapping to `@source <idx>`. |
| `Major` | Varint | 1-9 bytes | Package major version. |
| `Minor` | Varint | 1-9 bytes | Package minor version. |
| `Patch` | Varint | 1-9 bytes | Package patch version. |
| `HashCount`| Varint | 1+ bytes | Number of integrity hashes. |
| `Hashes` | Array | Variable| Array of hash structs (unchanged from v0.4). |
| `FeatureCount`| Varint | 1+ bytes | **NEW:** Number of strings in this package's local Feature Table. |
| `Features` | Array | Variable| **NEW:** Array of `(VarintLen, UTF-8 Bytes)` representing the features this package provides or requests. |
| `DepCount`| Varint | 1+ bytes | Number of direct dependencies. |
| `Dependencies`| Array | Variable| Array of dependency structs (see 4.2). |
| `CRC32` | Bytes | 4 bytes | Standard CRC32 (IEEE 802.3) checksum of all preceding bytes, little-endian. |

### 4.2 Dependency Struct Layout
Dependencies now point to a Content ID instead of a line index, and can specify requested features and platform targets.

| Field | Type | Size | Description |
| :--- | :--- | :--- | :--- |
| `ContentID`| Bytes | 8 bytes | FNV-1a 64-bit hash of `dep_name@dep_version` (little-endian). |
| `DepType` | Byte | 1 byte | Integer representing the dependency profile. |
| `TargetOS` | Byte | 1 byte | **Only present if `DepType == 0x04`**. Target OS enum. |
| `TargetArch`| Byte | 1 byte | **Only present if `DepType == 0x04`**. Target Arch enum. |
| `ReqFeatCount`| Varint | 1+ bytes | Number of requested features from the *local* Feature Table. |
| `ReqFeatureIdx`| Array | Variable| Array of Varints pointing to indices in the current package's `Features` array. (`0` means no features). |

### 4.3 Dependency Type Enum
*   `0x00`: **Runtime**
*   `0x01`: **Dev**
*   `0x02`: **Peer**
*   `0x03`: **Optional**
*   `0x04`: **OptionalTarget** (Requires `TargetOS` and `TargetArch` bytes immediately following).

### 4.4 Target Enums
**Target OS (`TargetOS`):**
*   `0x00`: Any/Unknown
*   `0x01`: Linux
*   `0x02`: MacOS
*   `0x03`: Windows

**Target Arch (`TargetArch`):**
*   `0x00`: Any/Unknown
*   `0x01`: x86_64 / AMD64
*   `0x02`: aarch64 / ARM64
*   `0x03`: wasm32

## 5. Public API Specification

### 5.1 Core Data Structures

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum TargetOS { Any, Linux, MacOS, Windows }

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum TargetArch { Any, X86_64, Aarch64, Wasm32 }

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum DepType {
    Runtime,
    Dev,
    Peer,
    Optional,
    OptionalTarget(TargetOS, TargetArch),
}

#[derive(Debug, Clone)]
pub struct Dependency {
    pub name: String,
    pub dep_type: DepType,
    pub requested_features: Vec<String>, // NEW
}

#[derive(Debug, Clone)]
pub struct Package {
    pub name: String,
    pub source_idx: usize,
    pub major: u64,
    pub minor: u64,
    pub patch: u64,
    pub hashes: Vec<IntegrityHash>,
    pub features: Vec<String>, // NEW
    pub dependencies: Vec<Dependency>,
}
// Lockfile, Source, HashAlgorithm, IntegrityHash, Override remain structurally unchanged.
```

### 5.2 Error Handling Additions

```rust
pub enum Error {
    // ... existing errors ...
    
    #[error("Package '{package}' depends on '{content_id:08x}', which was not found in the lockfile")]
    MissingContentId { package: String, content_id: u64 },

    #[error("Package '{package}' requests feature index {idx}, but its feature table only has {count} entries")]
    InvalidFeatureIndex { package: String, idx: usize, count: usize },
}
```

### 5.3 Core Functions
*   `fn serialize(lockfile: &mut Lockfile) -> Result<String, Error>`
    *   Iterates packages to build a `HashMap<String, u64>` mapping `name@ver` to FNV-1a 64-bit Content IDs.
    *   For each package, builds a local feature table and maps requested feature strings to local indices.
    *   Validates that all `requested_features` exist in the dependency's feature table (or standard known features).
*   `fn deserialize(content: &str) -> Result<Lockfile, Error>`
    *   Parses packages, building a `HashMap<u64, (String, Vec<String>)>` mapping `ContentID -> (Name, Features)`.
    *   Resolves dependencies by looking up the Content ID, extracting the name, and mapping the requested feature indices back to strings.
