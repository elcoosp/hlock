# hlock v0.16.0 Release Specification

**"Trust But Verify"**

**Release Manager:** elcoosp
**Status:** Draft
**Target Date:** TBD
**Previous Release:** v0.15.0
**Edition:** Rust 2024

---

## 1. Overview

hlock v0.16.0 is a reliability and usability release. It fixes a silent data-integrity bug in the digest computation, introduces a CLI binary for direct consumption in CI/CD pipelines, removes dead code paths, deduplicates internal parsing logic, and adds key rotation and VEX directives for supply-chain security maturity.

---

## 2. Motivation

Three problems constrain hlock adoption today:

1. **Integrity guarantee is broken for lockfiles with licenses.** The BLAKE3 `@digest` is computed before `@license` lines are appended, meaning any lockfile declaring licenses has a digest that does not cover those declarations. This is a silent violation of the core security promise.

2. **No CLI means no CI adoption.** Every consumer must write Rust code. Tools like `cosign`, `syft`, `osv-scanner`, and `npm audit` are consumed primarily as CLI commands in CI pipelines. hlock cannot compete without one.

3. **Static trust roots create operational risk.** There is no mechanism to rotate compromised or expired root keys without re-signing every lockfile. This violates TUF §5.2 and makes hlock unsuitable for long-lived projects.

## 3. Changes

### 3.1 CRITICAL: Fix Digest Calculation Order

**File:** `src/lockfile/mod.rs`, function `serialize`

**Problem:** The BLAKE3 digest is computed over `out.as_bytes()` before `@license` lines are appended. The `@digest` line is then appended last. During `validate_digest`, everything up to `@digest` — including the `@license` lines — is hashed. The hashes never match when licenses are present.

**Current behavior (broken):**
```
@source 0 https://r.com/

pkg\tCAQA
@license lodash MIT           ← NOT covered by digest
@digest abc123...             ← computed over content WITHOUT @license
```

**Required behavior (fixed):**
```
@source 0 https://r.com/

pkg\tCAQA
@license lodash MIT           ← NOW covered by digest
@digest def456...             ← computed over content WITH @license
```

**Change:**

In `serialize()`, move the `blake3::hash()` call to after the `@license` loop and before the `@digest` format. The resulting serialization order becomes:

1. Header directives
2. Empty line separator
3. Package lines (`name\t<base64url-payload>`)
4. `@artifact` directives
5. `@patch` directives
6. `@provenance` directives
7. `@advisory` directives
8. `@license` directives
9. Compute BLAKE3 digest over all of the above
10. `@digest` directive

**Migration:** Lockfiles produced by v0.15.0 that contain `@license` lines will fail `validate_digest` under v0.16.0. This is correct — those digests were already invalid. Consumers must re-serialize to generate correct digests. Lockfiles without `@license` lines are unaffected.

**Acceptance criteria:**
- `serialize` → `deserialize` → `validate_digest` round-trips successfully for lockfiles with licenses
- `serialize` → `validate_digest` succeeds directly on the serialized string
- Tampering any byte before the `@digest` line causes `validate_digest` to return `Error::DigestMismatch`

### 3.2 CLI Binary

**New file:** `src/main.rs`
**Modified file:** `Cargo.toml`

Add a CLI binary named `hlock` with the following subcommands. Use `clap` with derive macros.

#### 3.2.1 `hlock verify <file>`

Verifies lockfile integrity and trust chain.

```
hlock verify <file> [--trusted-key <key-id>:<algo>:<hex-pubkey>]... [--time <epoch>]
```

Steps, in order:
1. Read file content
2. Call `validate_digest` — exit 1 on `DigestMismatch` or `DuplicateDigestDirective`
3. Call `verify_signature` with provided trusted keys — exit 1 on `UntrustedKey`, `SignatureExpired`, `VerificationFailed`, `MlDsaVerificationFailed`
4. If signatures are present and verified, call `validate_trust_chain` with `--time` (default: current epoch) — exit 1 on `MissingTrustRoot` or `TrustRootExpired`
5. Exit 0 on success

Output format (stdout):
```
✓ digest valid
✓ signature valid (2 signatures verified)
✓ trust chain valid
```

On failure:
```
✗ digest mismatch: expected abc123, got def456
```

Exit codes:
- 0: all checks pass
- 1: verification failure
- 2: file not found or parse error

#### 3.2.2 `hlock lint <file>`

Runs the default lint rule set against the lockfile.

```
hlock lint <file> [--rule <rule-name>]... [--severity <level>] [--format <text|json>]
```

Steps:
1. `deserialize` the file
2. If `--rule` is specified, run only those named rules; otherwise run `lint_default`
3. If `--severity` is specified, filter to that severity and above (error > warning > info)
4. Output findings

Text output (default):
```
ERROR  no-git-urls          pkg-git   Package 'pkg-git' uses a git URL source which is non-reproducible
WARN   no-sha1              pkg-old   Package 'pkg-old' uses deprecated SHA-1 hash
INFO   require-attestation  pkg-new   Package 'pkg-new' has no supply chain attestation
```

JSON output (`--format json`):
```json
{
  "findings": [
    {
      "rule": "no-git-urls",
      "severity": "error",
      "package": "pkg-git",
      "message": "Package 'pkg-git' uses a git URL source which is non-reproducible"
    }
  ],
  "error_count": 1,
  "warning_count": 1,
  "info_count": 1
}
```

Exit codes:
- 0: no errors (warnings/info allowed)
- 1: one or more error-severity findings
- 2: file not found or parse error

#### 3.2.3 `hlock diff <old-file> <new-file>`

Computes the diff between two lockfiles.

```
hlock diff <old-file> <new-file> [--format <text|json>]
```

Steps:
1. Deserialize both files
2. Call `diff_lockfiles`
3. Call `serialize_diff` with chosen format (default: text)
4. Print to stdout

Exit codes:
- 0: diff computed (even if there are changes)
- 1: diff computation error
- 2: file not found or parse error

#### 3.2.4 `hlock audit <file>`

Produces a vulnerability audit report.

```
hlock audit <file> [--format <text|json>]
```

Steps:
1. Deserialize the file
2. Call `lockfile.audit()`
3. Print report

Text output:
```
CRITICAL  lodash   CVE-2024-12345   <4.17.21   https://...
HIGH      express  CVE-2024-67890   <4.21.0    https://...
---
Total: 2 critical/high, 0 medium, 0 low, 0 info
```

JSON output: serialized `AuditReport` struct.

Exit codes:
- 0: no critical or high vulnerabilities
- 1: critical or high vulnerabilities found
- 2: file not found or parse error

#### 3.2.5 `hlock sbom <file>`

Generates an SBOM from the lockfile.

```
hlock sbom <file> --namespace <ns> [--format <spdx-json|cyclonedx-json>]
```

Steps:
1. Deserialize the file
2. Call `generate_sbom` with chosen format (default: spdx-json)
3. Print to stdout

Exit codes:
- 0: success
- 1: SBOM generation error
- 2: file not found or parse error

#### 3.2.6 `hlock sign <file> --key-id <id> --algorithm <ed25519|mldsa65> --private-key <hex> --expires <epoch>`

Signs a lockfile and appends a `@signature` directive.

```
hlock sign <file> --key-id <id> --algorithm <ed25519|mldsa65> --private-key <hex-or-file> --expires <epoch> [--in-place]
```

Steps:
1. Serialize the lockfile (re-serializes to ensure canonical form)
2. Call `sign_lockfile` with provided parameters
3. If `--in-place`, write signed content back to the file; otherwise print to stdout

`--private-key` accepts either a hex string or a file path prefixed with `@` (e.g., `@/path/to/key`).

Exit codes:
- 0: success
- 1: signing error
- 2: file not found or parse error

#### 3.2.7 `hlock graph <file> --root <pkg-name>... [--platform <os>-<arch>]`

Extracts a subgraph rooted at the given packages.

```
hlock graph <file> --root <pkg-name>... [--platform <os>-<arch>] [--output <file>]
```

Steps:
1. Deserialize the file
2. Compute content IDs for root package names via `fnv::calculate`
3. If `--platform` is given, call `extract_subgraph_platform`; otherwise `extract_subgraph`
4. Serialize the result
5. Write to `--output` file or stdout

Platform format: `<os>-<arch>` where os is one of `linux`, `macos`, `windows`, `freebsd`, `android`, `ios`, `any` and arch is one of `x86_64`, `aarch64`, `wasm32`, `arm`, `s390x`, `ppc64le`, `any`.

Exit codes:
- 0: success
- 1: extraction error (e.g., `RootContentIdMissing`, `NoPackagesForPlatform`)
- 2: file not found or parse error

#### 3.2.8 `hlock merge --base <file> --ours <file> --theirs <file> [--strategy <ours|theirs|fail>]`

Merges two lockfile branches against a common base.

```
hlock merge --base <file> --ours <file> --theirs <file> [--strategy <ours|theirs|fail>] [--output <file>]
```

Steps:
1. Deserialize all three files
2. Call `merge_lockfiles` with chosen strategy (default: fail)
3. Print conflicts to stderr
4. Serialize merged lockfile to `--output` or stdout

Exit codes:
- 0: merge successful, no conflicts
- 1: merge has conflicts (when strategy is not `fail`, the merged result is still produced)
- 2: merge failed (strategy is `fail` and conflicts exist)
- 3: file not found or parse error

### 3.4 Deduplicate Header Parsing

**Files:** `src/lockfile/header.rs`, `src/lazy.rs`

**Problem:** `scan_header()` in `lazy.rs` reimplements all directive parsing from `parse_header()` in `header.rs`. The two implementations can drift.

**Change:**

Refactor `LazyLockfile::scan` to call `header::parse_header` for the header portion, then build the package index from the remaining content.

```rust
impl LazyLockfile {
    pub fn scan(content: &str) -> Result<LazyLockfile, Error> {
        let (lockfile, pkg_start_offset) = header::parse_header(content)?;

        let header = LockfileHeader {
            sources: lockfile.sources,
            overrides: lockfile.overrides,
            features: lockfile.features,
            metadata: lockfile.metadata,
            workspace_root: lockfile.workspace_root,
            workspace_pkgs: lockfile.workspace_pkgs,
            hoist_boundaries: lockfile.hoist_boundaries,
            artifacts: lockfile.artifacts,
            patches: lockfile.patches,
        };

        let mut index = Vec::new();
        let mut provenance = Vec::new();
        // ... build index from content[pkg_start_offset..]
        // ... (existing index-building logic stays, only header parsing is removed)

        Ok(LazyLockfile {
            content: Arc::from(content),
            header,
            index,
            provenance,
        })
    }
}
```

Delete the `scan_header` function and the `classify_source` function from `lazy.rs` entirely. `LockfileHeader` gains the fields that `parse_header` already populates (mirrors, policies, trust_roots) or the header struct is extended to carry them.

**Acceptance criteria:**
- `scan_header` function no longer exists in `lazy.rs`
- `classify_source` function no longer exists in `lazy.rs`
- All existing `lazy` tests pass unchanged
- A new test confirms that `LazyLockfile::scan` and `deserialize` agree on header contents for the same input

### 3.6 Compute `DedupOpportunity.potential_saving_bytes`

**File:** `src/policy.rs`, function `dedup_opportunities`

**Current:** Always returns `0`.

**Change:** Estimate savings as the byte cost of duplicate package entries. Each duplicate package incurs: the Base64URL-encoded payload line, the `@provenance` line, and any `@license` line.

```rust
potential_saving_bytes: {
    let per_entry_estimate: usize = versions.iter().skip(1).map(|_| {
        // Average: name (20) + tab (1) + base64 payload (~120) + newline (1)
        // + provenance line (~80) + license line (~30) ≈ 232 bytes
        232
    }).sum();
    per_entry_estimate
},
```

For a more precise estimate, the function can measure the serialized length of the first version's line and multiply by `(count - 1)`.

**Acceptance criteria:**
- `potential_saving_bytes` is non-zero for packages with multiple versions
- `potential_saving_bytes` is 0 for packages with a single version
- A test asserts the value is positive for a lockfile with duplicate `lodash` entries

### 3.7 Remove Ed448 Support

**Files:** `src/signature.rs`, `src/lockfile/header.rs`, `src/lockfile/types.rs` (if referenced)

**Change:**

Remove the `Ed448` variant from `SignatureAlgorithm`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum SignatureAlgorithm {
    Ed25519 = 0x00,
    MlDsa65 = 0x02,
}
```

Remove all `Ed448` match arms in `sign_lockfile`, `verify_signature`, and `parse_signature_directive`. The algorithm ID `0x01` becomes unassigned. If a lockfile contains `@signature ... 01 ...`, parsing returns `UnsupportedSignatureAlgorithm { algo_id: 1 }`.

Remove the Ed448-related tests: `test_ed448_sign_not_yet_available` and `test_ed448_verify_not_yet_available`.

Remove the `"01"` match arm in `header.rs` `parse_header` for `@trust-root`. If a lockfile contains `@trust-root ... 01 ...`, parsing returns `InvalidHeader` with reason "Unknown algorithm: 01".

Update `format_header` to remove the `"01"` => `Ed448` arm.

**Migration:** No v0.15.0 lockfile can contain a valid Ed448 signature (signing was unimplemented), so no data migration is needed. The `0x01` algorithm ID is reserved for future use.

**Acceptance criteria:**
- No `Ed448` variant exists in `SignatureAlgorithm`
- No `Ed448` match arms in any source file
- Algorithm ID `0x01` is rejected with `UnsupportedSignatureAlgorithm`
- All remaining tests pass

### 3.8 Add `@trust-root-rotation` Directive

**File:** `src/lockfile/types.rs`, `src/lockfile/header.rs`, `src/policy.rs`

**New type:**

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct TrustRootRotation {
    pub old_key_id: String,
    pub new_key_id: String,
    pub threshold: u8,
    pub new_algorithm: SignatureAlgorithm,
    pub new_public_key: Vec<u8>,
    pub new_expires_epoch: u64,
    pub new_role: TrustRole,
    pub rotation_signature_key_id: String,
    pub rotation_signature: Vec<u8>,
}
```

**Wire format:**
```
@trust-root-rotation <old-key-id> <new-key-id> <threshold> <algo-hex> <new-pubkey-hex> <new-expires> <new-role> <sig-key-id> <sig-hex>
```

Example:
```
@trust-root-rotation old@key.com new@key.com 1 00 2a2a...2a 1800000000 root old@key.com a1b2c3...f4
```

**Parsing (in `header.rs`):** Add a `strip_prefix("@trust-root-rotation ")` arm that parses the 9 fields.

**Serialization (in `format_header`):** Emit `@trust-root-rotation` lines after `@trust-root` lines.

**Validation (new method on `Lockfile`):**

```rust
pub fn validate_root_rotation(&self, now_epoch: u64) -> Result<(), Error> {
    for rotation in &self.root_rotations {
        // 1. old_key_id must exist in trust_roots with role Root
        let old_root = self.trust_roots.iter()
            .find(|t| t.key_id == rotation.old_key_id && t.role == TrustRole::Root)
            .ok_or(Error::TrustRootRotationInvalid {
                reason: format!("old key '{}' not found as root", rotation.old_key_id),
            })?;

        // 2. Verify the rotation signature was made by the old key
        //    over the new key material
        let rotation_message = format!(
            "{}:{}:{}:{}:{}",
            rotation.new_key_id,
            rotation.threshold,
            match rotation.new_algorithm {
                SignatureAlgorithm::Ed25519 => "00",
                SignatureAlgorithm::MlDsa65 => "02",
            },
            rotation.new_public_key.iter().map(|b| format!("{:02x}", b)).collect::<String>(),
            rotation.new_expires_epoch,
        );

        // 3. The old root key must not be expired at the time of rotation
        if old_root.expires_epoch != 0 && old_root.expires_epoch < now_epoch {
            return Err(Error::TrustRootExpired {
                key_id: old_root.key_id.clone(),
                expires_epoch: old_root.expires_epoch,
            });
        }

        // 4. The new key must not already be expired
        if rotation.new_expires_epoch != 0 && rotation.new_expires_epoch < now_epoch {
            return Err(Error::TrustRootExpired {
                key_id: rotation.new_key_id.clone(),
                expires_epoch: rotation.new_expires_epoch,
            });
        }
    }
    Ok(())
}
```

**New error variant:**
```rust
#[error("Trust root rotation invalid: {reason}")]
TrustRootRotationInvalid { reason: String },
```

**`Lockfile` struct gains:**
```rust
pub root_rotations: Vec<TrustRootRotation>,
```

**Acceptance criteria:**
- `@trust-root-rotation` round-trips through serialize/deserialize
- `validate_root_rotation` rejects rotations from non-root keys
- `validate_root_rotation` rejects rotations from expired keys
- `validate_root_rotation` accepts valid rotations
- Rotation signature verification uses the old root's public key and algorithm

### 3.9 Add `@vex` Directive

**File:** `src/lockfile/types.rs`, `src/lockfile/header.rs` (or `mod.rs`), `src/policy.rs`

**New types:**

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum VexStatus {
    NotAffected,
    Affected,
    Fixed,
    UnderInvestigation,
}

impl VexStatus {
    pub fn from_str(s: &str) -> Option<Self> {
        match s {
            "not_affected" => Some(VexStatus::NotAffected),
            "affected" => Some(VexStatus::Affected),
            "fixed" => Some(VexStatus::Fixed),
            "under_investigation" => Some(VexStatus::UnderInvestigation),
            _ => None,
        }
    }

    pub fn as_str(&self) -> &'static str {
        match self {
            VexStatus::NotAffected => "not_affected",
            VexStatus::Affected => "affected",
            VexStatus::Fixed => "fixed",
            VexStatus::UnderInvestigation => "under_investigation",
        }
    }
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct VexEntry {
    pub package: String,
    pub advisory_id: String,
    pub status: VexStatus,
    pub justification: String,
    pub impact_statement: String,
}
```

**Wire format:**
```
@vex <package> <advisory-id> <status> <justification> <impact-statement>
```

Fields `justification` and `impact_statement` may contain spaces; they are the 4th and 5th space-separated fields parsed with `splitn(5, ' ')`.

Example:
```
@vex lodash CVE-2024-12345 not_affected vulnerable_code_not_in_execute_path not_applicable
```

**Parsing:** After the empty line separator, handle `@vex` prefixed lines similarly to `@advisory`.

**Serialization:** After `@advisory` lines, before `@license` lines (and thus before the digest computation).

**New error variant:**
```rust
#[error("Line {line_number}: Invalid VEX status '{status}'")]
InvalidVexStatus { line_number: usize, status: String },
```

**New method on `Lockfile`:**

```rust
pub fn vex_for(&self, package_name: &str, advisory_id: &str) -> Option<&VexEntry> {
    self.vex_entries.iter()
        .find(|v| v.package == package_name && v.advisory_id == advisory_id)
}

pub fn effective_advisories(&self) -> AuditReport {
    let report = self.audit();
    let mut filtered = AuditReport::default();
    for adv in report.all_advisories() {
        let vex = self.vex_for(&adv.package, &adv.advisory_id);
        match vex.map(|v| &v.status) {
            Some(VexStatus::NotAffected) | Some(VexStatus::Fixed) => continue,
            _ => {}
        }
        match adv.severity {
            AdvisorySeverity::Critical => filtered.critical.push(adv.clone()),
            AdvisorySeverity::High => filtered.high.push(adv.clone()),
            AdvisorySeverity::Medium => filtered.medium.push(adv.clone()),
            AdvisorySeverity::Low => filtered.low.push(adv.clone()),
            AdvisorySeverity::Info => filtered.info.push(adv.clone()),
        }
    }
    filtered
}
```

**`Lockfile` struct gains:**
```rust
pub vex_entries: Vec<VexEntry>,
```

**Acceptance criteria:**
- `@vex` round-trips through serialize/deserialize
- Invalid VEX status strings produce `InvalidVexStatus` error
- `vex_for` returns the correct entry
- `effective_advisories` excludes `not_affected` and `fixed` entries
- `effective_advisories` includes `affected` and `under_investigation` entries
- `effective_advisories` includes advisories with no VEX entry
- VEX entries are covered by the BLAKE3 digest

| `@vex` | Post-header | `@vex lodash CVE-2024-12345 not_affected code_not_present not_applicable` |

### 6.2 Modified Directives

| Directive | Change |
|---|---|
| `@trust-root ... 01 ...` | Algorithm ID `01` is now rejected as unknown |

### 6.3 Digest Coverage Change

The `@digest` now covers `@license` lines. This is a breaking change for lockfiles that contain `@license` directives.

### 6.4 Canonical Serialization Order (v0.16.0)

```
Header section:
  @source
  @mirror
  @policy
  @trust-root
  @trust-root-rotation     ← NEW
  @override
  @feature
  @workspace-root
  @workspace-pkg
  @hoist-boundary
  @metadata
  <empty line>

Package section:
  <name>\t<base64url-payload>    (sorted by name)
  @artifact
  @patch
  @provenance
  @advisory
  @vex                         ← NEW
  @license
  @digest                      ← now covers @license
  @signature                   (if present)
```

## 7. Migration Guide (v0.15.0 → v0.16.0)

### 7.1 Digest Recomputation

Lockfiles with `@license` directives must be re-serialized. The old digest does not cover license data.

```rust
let mut lockfile = hlock::read_lockfile(path)?;
hlock::write_lockfile(path, &mut lockfile)?; // re-serializes with correct digest
```

Or via CLI:
```bash
hlock verify old.hlock   # will report DigestMismatch
# Re-serialize:
# (future: hlock rehash old.hlock --in-place)
```

### 7.2 Ed448 Removal

No v0.15.0 lockfile can contain a valid Ed448 signature (signing was unimplemented). If any lockfile has a hand-crafted `@signature ... 01 ...` line, it will be rejected with `UnsupportedSignatureAlgorithm`. Remove such lines.

### 7.3 New `Lockfile` Fields

The `Lockfile` struct has two new fields. Code that constructs `Lockfile` manually must add:

```rust
root_rotations: vec![],
vex_entries: vec![],
```

If using `..Lockfile::default()`, no change is needed (the `Default` derive handles it).
