# HLOCK Specification v0.10.0

## 1. Overview
HLOCK v0.10.0 is **"The Monorepo & Build Integrity Release"**. It addresses the two remaining sources of non-reproducibility in modern package management: lifecycle script tampering and untracked patches, while introducing explicit workspace topology to make lockfiles fully self-contained for monorepo environments.

### Design Principles
- **Hermetic Build Context** — A single `.hlock` file encodes not just *what code* is installed (hashes) but *what happens next* (lifecycle script hashes). An installer can verify the full build pipeline before executing a single script.
- **Self-Contained Monorepo** — Workspace topology, hoist boundaries, and patch manifests are declared inside the lockfile. No filesystem crawling for `package.json` files is needed to understand the monorepo layout.
- **Patch Integrity** — Patches applied to dependencies are tracked with cryptographic digests, closing a supply chain gap where a patched lockfile could pass CRC checks while the actual patch content drifted.

**Backward Compatibility:** v0.10 is fully backward compatible with v0.9 lockfiles. The payload version byte remains `0x06`. New binary payload sections are appended after the v0.9 `PlatformTags` array and before the `CRC32` trailer. New header directives are ignored by v0.9 parsers.

---

## 2. File Structure

```text
<header directives>
<empty line>
<package lines>
<optional patch directives>
<optional signature directive>
```

The structural changes from v0.9 are:
1. New header directives (`@workspace-root`, `@workspace-pkg`, `@hoist-boundary`) appear before the empty-line separator, after existing `@source`/`@override`/`@feature` directives.
2. New `@patch` directives appear after all package lines but before any `@signature` directive.

---

## 3. Header Directives

All v0.9 header directives are unchanged:

```text
@source <idx> <uri>
@override <name> <ver> -> <ver>
@feature <name> [flag1,flag2,...]
```

### 3.1 New: Workspace Topology Directives

```text
@workspace-root <path>
@workspace-pkg <name> <manifest_path>
@hoist-boundary <consumer> -> [dep1,dep2,...]
```

#### `@workspace-root <path>`

Declares the root directory of the workspace relative to the lockfile location. Must appear exactly once if any `@workspace-pkg` directives are present.

| Field | Description |
|---|---|
| `path` | Relative path to the workspace root (e.g., `.`, `..`, `../../monorepo`). |

#### `@workspace-pkg <name> <manifest_path>`

Declares a workspace member package and the path to its manifest file.

| Field | Description |
|---|---|
| `name` | The canonical package name (must match a package line in the lockfile). |
| `manifest_path` | Relative path from the workspace root to the manifest file (e.g., `apps/web/package.json`). |

**Rules:**
- Every package with `Source::Workspace` SHOULD have a corresponding `@workspace-pkg` directive.
- The `manifest_path` is informational for tooling; it is not used in binary payload encoding.

#### `@hoist-boundary <consumer> -> [dep1,dep2,...]`

Declares that `consumer` is only allowed to resolve the listed dependencies (and their transitive closures) from its direct node_modules. Any dependency NOT in the list becomes a "phantom dependency" error if resolved outside the boundary.

| Field | Description |
|---|---|
| `consumer` | A workspace package name. |
| `dep1,dep2,...` | Comma-separated list of allowed direct dependency names. An empty list `[]` means no external dependencies are allowed. |

**Rules:**
- `consumer` MUST have a corresponding `@workspace-pkg` directive.
- If no `@hoist-boundary` is present for a workspace package, all dependencies are allowed (legacy permissive mode).
- Installers SHOULD warn or error when a phantom dependency is detected during installation.
- The boundary list is checked against the *direct* dependencies of the consumer, not transitive dependencies. Transitive dependencies are always allowed.

**Example:**
```text
@workspace-root .
@workspace-pkg apps/web apps/web/package.json
@workspace-pkg packages/core packages/core/package.json
@workspace-pkg packages/utils packages/utils/package.json
@hoist-boundary apps/web -> [packages/core,packages/utils]
```

### 3.2 New: Patch Directive

```text
@patch <content_id_hex> <patch_type> <relative_path>
```

Declares that a patch file has been applied to a specific locked package.

| Field | Description |
|---|---|
| `content_id_hex` | The FNV-1a 64-bit content ID of the patched package, as a 16-character lowercase hex string (e.g., `8f14e45fceea167a`). |
| `patch_type` | `0x01` for a unified diff. (Reserved for future types.) |
| `relative_path` | Relative path from the lockfile location to the patch file. |

**Rules:**
- There MUST be at most one `@patch` directive per content ID.
- `@patch` directives MUST appear after all package lines and before any `@signature` directive.
- The patch file itself is NOT embedded in the lockfile; only its path is recorded. The integrity of the patch is tracked in the binary payload (see §5.4).
- v0.9 and earlier parsers MUST ignore `@patch` directives without error.

**Example:**
```text
@patch 8f14e45fceea167a 0x01 patches/fix-core-overflow.diff
```

## 4. Content IDs

Unchanged from v0.9. Dependencies are still referenced by FNV-1a 64-bit hashes of `canonical_name@major.minor.patch`.

## 5. The Binary Payload (v0.10 Schema)

### 5.1 Payload Layout

The payload version byte remains `0x06`. v0.10 appends two new sections after the v0.9 `PlatformTags` array and before the `CRC32` trailer.

| Field | Type | Size | Description |
|---|---|---|---|
| `Version` | Byte | 1 byte | `0x06` (unchanged) |
| `LogicalNameLen` | Varint | 1+ bytes | (unchanged) |
| `LogicalName` | Bytes | Variable | (unchanged) |
| `SourceIdx` | Varint | 1+ bytes | (unchanged) |
| `Major` | Varint | 1-9 bytes | (unchanged) |
| `Minor` | Varint | 1-9 bytes | (unchanged) |
| `Patch` | Varint | 1-9 bytes | (unchanged) |
| `HashCount` | Varint | 1+ bytes | (unchanged) |
| `Hashes` | Array | Variable | (unchanged) |
| `FeatureCount` | Varint | 1+ bytes | (unchanged) |
| `Features` | Array | Variable | (unchanged) |
| `PeerCount` | Varint | 1+ bytes | (unchanged) |
| `Peers` | Array | Variable | (unchanged) |
| `DepCount` | Varint | 1+ bytes | (unchanged) |
| `Dependencies` | Array | Variable | (unchanged) |
| `PeerReqCount` | Varint | 1+ bytes | (unchanged) |
| `PeerRequirements` | Array | Variable | (unchanged) |
| `PlatformTagCount` | Varint | 1+ bytes | (unchanged) |
| `PlatformTags` | Array | Variable | (unchanged) |
| `ScriptHashCount` | Varint | 1+ bytes | **NEW** |
| `ScriptHashes` | Array | Variable | **NEW** (see §5.2) |
| `PatchHash` | Bytes | 0 or 33 bytes | **NEW** (see §5.3) |
| `CRC32` | Bytes | 4 bytes LE | Checksum of all preceding bytes (unchanged scope) |

**Backward Compatibility:** Same strategy as v0.9. The `deserialize` function auto-detects: after reading `PlatformTags`, if the remaining bytes are exactly 4 (`CRC32`), parse as v0.9 or earlier. If more than 4 bytes remain, parse the v0.10 sections.

The `CompatMode` enum gains a new variant:

```text
enum CompatMode {
    V8,    // Omit v0.9 and v0.10 sections
    V9,    // Include v0.9 sections, omit v0.10 sections
    V10,   // Include v0.9 and v0.10 sections
}
```

The `serialize` function defaults to `CompatMode::V10`.

### 5.2 Script Hash Struct Layout

Declares a cryptographic hash of a lifecycle script defined in the package's manifest (e.g., `postinstall`, `prepare`, `prebuild`).

| Field | Type | Size | Description |
|---|---|---|---|
| `ScriptType` | Byte | 1 byte | Script identifier (see table below). |
| `HashAlgo` | Byte | 1 byte | Hash algorithm ID (same as §5 Hashes: 0x01=SHA-256, 0x03=BLAKE3). |
| `DigestLen` | Byte | 1 byte | Length of the digest in bytes. |
| `Digest` | Bytes | Variable | The hash digest. |

**Script Type IDs:**

| ID | Script |
|---|---|
| 0x00 | `prepare` |
| 0x01 | `preinstall` |
| 0x02 | `install` |
| 0x03 | `postinstall` |
| 0x04 | `prebuild` |
| 0x05 | `build` |
| 0x06 | `postbuild` |
| 0xFF | Other (unknown, informational only) |

**Semantics:**
- The digest is computed over the **script source text** as it appears in the manifest (e.g., the value of the `"scripts"."postinstall"` field in `package.json`). Shell variable expansion, `$(...)` subshells, and environment variable references are NOT expanded before hashing — the raw text is hashed.
- If a script's content changes, the digest changes, and the lockfile CRC fails. This forces the lockfile to be regenerated.
- If a script referenced by a `ScriptHash` entry is missing from the manifest, installers SHOULD warn but NOT fail (the script may have been removed intentionally).
- If a manifest has scripts that are NOT covered by `ScriptHash` entries, installers SHOULD warn (unhashed scripts are a reproducibility risk).
- Empty `ScriptHashCount` (0x00) means no scripts are tracked (default, backward compatible).

**Example:** A `postinstall` script with BLAKE3 hash:
- `ScriptType` = `0x03`
- `HashAlgo` = `0x03`
- `DigestLen` = `0x20` (32)
- `Digest` = 32 bytes of BLAKE3 digest

### 5.3 Patch Hash

A single optional field appended after the `ScriptHashes` array. If present, it is exactly 33 bytes. If absent, it is 0 bytes.

| Field | Type | Size | Description |
|---|---|---|---|
| `PatchHashPresent` | Byte | 1 byte | `0x01` if a patch hash follows, `0x00` if no patch (default). |
| `HashAlgo` | Byte | 1 byte | Hash algorithm ID (same as §5 Hashes). Only present if `PatchHashPresent` is `0x01`. |
| `Digest` | Bytes | 32 bytes | BLAKE3 digest of the patch file content. Only present if `PatchHashPresent` is `0x01`. |

**Semantics:**
- If a package has a corresponding `@patch` directive in the lockfile header, its binary payload MUST include a `PatchHash`.
- The digest covers the raw bytes of the patch file (as read from disk), NOT the patched source code.
- Installers SHOULD verify that the patch file at the declared path matches this digest before applying it.
- If `PatchHashPresent` is `0x01` but no `@patch` directive exists for this package, the lockfile is malformed and parsers MUST return an error.

## 6. Public API Specification

### 6.1 New Data Structures

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct ScriptHash {
    pub script_type: ScriptType,
    pub hash_algo: HashAlgorithm,
    pub digest: Vec<u8>,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum ScriptType {
    Prepare,
    PreInstall,
    Install,
    PostInstall,
    PreBuild,
    Build,
    PostBuild,
    Other,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct WorkspacePkg {
    pub name: String,
    pub manifest_path: String,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct HoistBoundary {
    pub consumer: String,
    pub allowed_deps: Vec<String>,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct PatchDirective {
    pub content_id: u64,
    pub patch_type: u8,
    pub relative_path: String,
}
```

### 6.2 Changes to Existing Structures

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Package {
    // ... all existing fields ...
    pub peer_requirements: Vec<PeerRequirement>,
    pub platform_tags: Vec<PlatformTag>,
    pub script_hashes: Vec<ScriptHash>,    // NEW
    pub patch_hash: Option<(HashAlgorithm, Vec<u8>)>,  // NEW
}

#[derive(Debug, Clone)]
pub struct Lockfile {
    pub sources: Vec<Source>,
    pub overrides: Vec<Override>,
    pub features: Vec<(String, Vec<String>)>,
    pub workspace_root: Option<String>,       // NEW
    pub workspace_pkgs: Vec<WorkspacePkg>,     // NEW
    pub hoist_boundaries: Vec<HoistBoundary>,  // NEW
    pub packages: Vec<Package>,
    pub patches: Vec<PatchDirective>,         // NEW
}

pub enum CompatMode {
    V8,
    V9,
    V10,  // NEW
}
```

### 6.3 New Functions

```rust
/// Validate hoist boundaries for a workspace package.
/// Returns Ok(()) if all direct dependencies of `consumer` are in the allowed list.
/// Returns Err if a phantom dependency is detected.
pub fn validate_hoist_boundary(
    lockfile: &Lockfile,
    consumer_name: &str,
) -> Result<(), Error>;

/// Validate all patch hashes in the lockfile against files on disk.
/// Returns Ok(()) if all patch files match their digests.
/// Returns Err if a patch file is missing or has a different digest.
pub fn validate_patches(
    lockfile: &Lockfile,
    lockfile_dir: &std::path::Path,
) -> Result<(), Error>;

/// Validate all script hashes against manifest files on disk.
/// Returns Ok(()) if all hashed scripts match their digests.
/// Returns a list of warnings for unhashed scripts.
pub fn validate_scripts(
    lockfile: &Lockfile,
    lockfile_dir: &std::path::Path,
) -> Result<Vec<String>, Error>;
```

### 6.4 New Error Variants

```rust
#[derive(Debug, thiserror::Error)]
pub enum Error {
    // ... existing variants ...

    #[error("Phantom dependency: '{dep}' is used by '{consumer}' but not in its hoist boundary")]
    PhantomDependency { consumer: String, dep: String },

    #[error("Patch file for '{package}' (content ID {content_id:016x}) not found at '{path}'")]
    PatchFileMissing { package: String, content_id: u64, path: String },

    #[error("Patch file for '{package}' has mismatched digest: expected {expected}, got {actual}")]
    PatchDigestMismatch { package: String, expected: String, actual: String },

    #[error("Package '{package}' has a patch hash but no @patch directive in the lockfile header")]
    OrphanPatchHash { package: String },

    #[error("Script '{script}' for package '{package}' has mismatched digest")]
    ScriptDigestMismatch { package: String, script: String },
}
```

### 6.5 Parser Behavior

- `deserialize`: After reading `PlatformTags`, check remaining bytes. If `cursor + 4 == bytes.len()`, parse as v0.9. If more bytes remain, parse `ScriptHashCount`, `ScriptHashes`, `PatchHash` (detected by reading one byte: `0x00` = absent, `0x01` = present + 33 bytes), then `CRC32` as v0.10.
- `validate_hoist_boundary`: Look up the consumer package's direct dependencies. Cross-reference with the `HoistBoundary.allowed_deps` list. If any direct dependency is not in the list, return `PhantomDependency`.
- `validate_patches`: For each `PatchDirective`, read the file from `lockfile_dir.join(relative_path)`, compute the digest, and compare to the `patch_hash` in the corresponding package's binary payload.
- `validate_scripts`: For each package with `ScriptHash` entries, attempt to read the manifest from `lockfile_dir.join(workspace_root).join(manifest_path)` (if workspace) or the package's source, extract the script text, compute the digest, and compare. Collect warnings for packages that have scripts in their manifest but no corresponding `ScriptHash` entries.

## 7. Implementation Notes for Package Managers

### 7.1 Hermetic Build Pipeline

A CI pipeline using v0.10 lockfiles achieves full hermeticity:

```text
1. Parse lockfile (deserialize).
2. verify_signature(...).     # v0.9: who produced this?
3. validate_patches(...).     # v0.10: were patches tampered with?
4. validate_scripts(...).     # v0.10: were build scripts tampered with?
5. Install packages.
6. Run build.
```

No step after #1 requires network access or filesystem trust.

### 7.2 Phantom Dependency Enforcement

In strict mode, the installer:

```text
for each workspace package with a @hoist-boundary:
    for each direct dependency:
        if dep not in allowed_deps:
            error("phantom dependency detected")
```

This prevents the #1 source of "works on my machine but fails in CI" bugs in monorepos with hoisted node_modules.

### 7.3 Patch Workflow

```text
# Developer applies a patch:
pnpm patch packages/core@1.0.0
# Edit node_modules/.pnpm_patches/...
pnpm patch-commit packages/core@1.0.0

# Lockfile generator records the patch:
@patch 8f14e45fceea167a 0x01 patches/fix-core-overflow.diff

# In CI, before install:
hlock validate-patches hlock.lock

# On install, the installer:
# 1. Reads patches/fix-core-overflow.diff
# 2. Verifies BLAKE3 matches the patch_hash in the binary payload
# 3. Applies the patch
```

### 7.4 Script Hash Workflow

```text
# Lockfile generator reads package.json scripts:
{
  "scripts": {
    "postinstall": "node download-binary.js",
    "prepare": "npm run build"
  }
}

# Computes BLAKE3 of "node download-binary.js" and "npm run build"
# Encodes as ScriptHash entries in the binary payload

# In CI:
hlock validate-scripts hlock.lock
# Output:
# WARNING: package 'foo' has unhashed script 'prebuild'
# ERROR: package 'bar' script 'postinstall' digest mismatch
```

### 7.5 CompatMode Behavior

| Mode | Peer Reqs | Platform Tags | Script Hashes | Patch Hash |
|---|---|---|---|---|
| `V8` | ✗ | ✗ | ✗ | ✗ |
| `V9` | ✓ | ✓ | ✗ | ✗ |
| `V10` | ✓ | ✓ | ✓ | ✓ |

When serializing in `V8` or `V9` mode, any packages with `script_hashes` or `patch_hash` data MUST emit a warning. The data is silently discarded.

## 8. Migration Guide (v0.9 → v0.10)

| Area | Action |
|---|---|
| Existing lockfiles | No migration needed. v0.10 deserializers read v0.9 (and v0.8) payloads transparently. |
| Serializers | Default to `CompatMode::V10`. Add `--compat v9` and `--compat v8` CLI flags if needed. |
| Package structures | Initialize `script_hashes: vec![]` and `patch_hash: None` on all existing packages. |
| Lockfile structures | Initialize `workspace_root: None`, `workspace_pkgs: vec![]`, `hoist_boundaries: vec![]`, `patches: vec![]`. |
| Subgraph extraction | `extract_subgraph` and `extract_subgraph_platform` are unchanged. Workspace topology directives are preserved in the extracted lockfile's header. |
| Script hash validation | Optional. Add to CI pipelines incrementally. Lockfiles without script hashes are still valid. |
| Patch validation | Optional. Add to CI pipelines incrementally. Lockfiles without patches are still valid. |
