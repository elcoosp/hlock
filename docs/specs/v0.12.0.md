# HLOCK Specification v0.12.0

## 1. Overview

HLOCK v0.12.0 is **"The Cryptographic Agility & Graph Intelligence Release"**. It replaces the CRC32 payload trailer with a BLAKE3 digest (eliminating the last non-cryptographic integrity mechanism), introduces a pluggable signature algorithm, adds graph query primitives for reverse-dependency resolution and cycle detection, and formalizes multi-signature semantics.

This is a hard breaking change from v0.11. The payload version byte advances from `0x07` to `0x08`. v0.11 lockfiles cannot be read by a v0.12 parser.

### Design Principles

- **No Weak Links** — Every integrity check in the format is now cryptographic. CRC32 is gone. The only remaining question is *which* hash, not *whether* to hash.
- **Algorithm Agility** — Signature algorithms and hash algorithms are identified by IDs, not hardcoded. Adding post-quantum signatures (e.g., ML-DSA) in v0.13 requires only a new ID, not a new format.
- **Graph as a First-Class Citizen** — The lockfile is not just a serialization format; it is a queryable dependency graph. Consumers should never need to reimplement topological sorts or reverse lookups.
- **Multi-Signature, Single Policy** — Multiple `@signature` directives are allowed. The verification policy is simple: **all signatures must verify**. Policy complexity (threshold, per-package rules) is the verifier's responsibility, not the format's.

## 3. Header Directives

### 3.1 Unchanged Directives

```text
@source <idx> <uri>
@override <name> <ver> -> <ver>
@feature <name> [flag1,flag2,...]
@workspace-root <path>
@workspace-pkg <name> <manifest_path>
@hoist-boundary <consumer> -> [dep1,dep2,...]
@metadata <key> <value>
@artifact <content_id_hex> <os_id> <arch_id> <relative_path>
@patch <content_id_hex> <patch_type> <relative_path>
```

### 3.2 Signature Directive (Changed)

```text
@signature <key_id> <algo_id> <expires_epoch> <base64_sig>
```

| Field | Description |
|---|---|
| `key_id` | Identifier for the signing key (no spaces). |
| `algo_id` | Single byte encoded as two hex digits. `00` = Ed25519, `01` = Ed448. |
| `expires_epoch` | Unix timestamp (decimal). `0` means no expiration. Verifiers MUST reject signatures with non-zero expiration where the current time exceeds the expiration. |
| `base64_sig` | Base64URL-encoded signature bytes. Ed25519 = 64 bytes, Ed448 = 114 bytes. |

**Rules:**
- Multiple `@signature` directives are allowed.
- All `@signature` directives MUST appear after all package, `@artifact`, and `@patch` lines.
- The signature covers all bytes from the start of the file to the character *before* the `@` of the first `@signature` line.
- Verifiers MUST verify ALL `@signature` directives. If any fails verification (crypto failure, expired key, malformed encoding), the entire lockfile is rejected.
- If no `@signature` directives are present, the lockfile is valid but unsigned. Verifiers MAY treat unsigned lockfiles as untrusted.

**Migration from v0.11:**

| v0.11 | v0.12 |
|---|---|
| `@signature ci@co.com <sig>` | `@signature ci@co.com 00 0 <sig>` |

The v0.11 format implicitly used Ed25519 with no expiration. v0.12 makes both explicit.

## 5. The Binary Payload (v0.12 Schema)

### 5.1 Payload Layout

The version byte is `0x08`. The layout is identical to v0.11 except the trailer.

| Field | Type | Size | Description |
|---|---|---|---|
| `Version` | Byte | 1 byte | `0x08` |
| `LogicalNameLen` | Varint | 1+ bytes | Length of logical name, or `0x00` if none. |
| `LogicalName` | Bytes | Variable | The alias name. |
| `SourceIdx` | Varint | 1+ bytes | Index into the `@source` array. |
| `Major` | Varint | 1-9 bytes | |
| `Minor` | Varint | 1-9 bytes | |
| `Patch` | Varint | 1-9 bytes | |
| `HashCount` | Varint | 1+ bytes | |
| `Hashes` | Array | Variable | Integrity hashes of the package source. |
| `FeatureCount` | Varint | 1+ bytes | |
| `Features` | Array | Variable | Enabled feature flags. |
| `PeerCount` | Varint | 1+ bytes | |
| `Peers` | Array | Variable | Resolved peer dependencies. |
| `DepCount` | Varint | 1+ bytes | |
| `Dependencies` | Array | Variable | Standard dependencies. |
| `PeerReqCount` | Varint | 1+ bytes | |
| `PeerRequirements` | Array | Variable | Unresolved peer requirements. |
| `PlatformTagCount`| Varint | 1+ bytes | |
| `PlatformTags` | Array | Variable | Target platform constraints. |
| `ExportCount` | Varint | 1+ bytes | |
| `Exports` | Array | Variable | Public interface map. |
| `ArtifactCount` | Varint | 1+ bytes | |
| `Artifacts` | Array | Variable | Platform-specific binary hashes. |
| `HookHashCount` | Varint | 1+ bytes | |
| `HookHashes` | Array | Variable | Lifecycle hook hashes. |
| `PatchHash` | Bytes | 0 or 33 bytes | |
| **`BLAKE3`** | **Bytes** | **32 bytes** | **Replaces CRC32. First 32 bytes of BLAKE3 hash of all preceding bytes.** |

### 5.2 Why BLAKE3 Replaces CRC32

CRC32 is a 32-bit error-detecting code with no cryptographic properties. It has roughly 1-in-4-billion collision probability — trivially forgeable. In a supply-chain security context, an attacker who can modify the Base64URL payload can find a CRC32 collision in seconds and produce a payload that passes integrity checking but contains malicious data.

BLAKE3 is:
- **128-bit collision resistance** (256-bit preimage resistance).
- **Faster than CRC32** on modern hardware with SIMD.
- **Already a dependency** (used as `HashAlgorithm::Blake3` for package source hashes).
- **Deterministic and parallelizable**.

The 28-byte overhead per package (32 bytes vs 4 bytes) is negligible compared to typical digest sizes. A package with a single SHA-256 integrity hash already carries 32 bytes of digest data.

### 5.3 Struct Layouts (Unchanged from v0.11)

All inner struct layouts — Hash, Export, Artifact, HookHash, Dependency, PeerRequirement, PlatformTag — are unchanged from v0.11. No field ordering, sizing, or encoding changes.

## 6. Public API Specification

### 6.1 Removed API

- `fn pack_payload(...)` — Replaced by `fn pack_payload_v12(...)`. (The old function is renamed internally; the public API surface gains the explicit version suffix for clarity, though only one version exists.)
- `fn unpack_payload(...)` — Replaced by `fn unpack_payload_v12(...)`.

### 6.2 New Data Structures

```rust
/// Identifies the signature algorithm used in @signature directives.
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum SignatureAlgorithm {
    Ed25519 = 0x00,
    Ed448 = 0x01,
}

/// Parsed representation of an @signature directive.
#[derive(Debug, Clone)]
pub struct SignatureDirective {
    pub key_id: String,
    pub algorithm: SignatureAlgorithm,
    pub expires_epoch: u64,  // 0 = no expiration
    pub signature_bytes: Vec<u8>,
}
```

### 6.3 Changes to `verify_signature`

```rust
/// Verifies ALL @signature directives in the lockfile content.
///
/// Returns Ok(()) if:
/// - No @signature directives exist (unsigned lockfile), OR
/// - All @signature directives verify with their respective algorithms
///   and have not expired.
///
/// Returns Err if ANY directive fails verification, is malformed,
/// or has expired.
pub fn verify_signature(
    lockfile_content: &str,
    trusted_keys: &HashMap<String, (&[u8], SignatureAlgorithm)>,
) -> Result<(), SignatureError>
```

The `trusted_keys` map associates `key_id` strings to `(public_key_bytes, expected_algorithm)` pairs. The verifier:
1. Parses all `@signature` directives.
2. For each directive, looks up `key_id` in `trusted_keys`.
3. If the key is not in the trusted set, returns `SignatureError::UntrustedKey { key_id }`.
4. Checks expiration. If `expires_epoch != 0` and `current_time > expires_epoch`, returns `SignatureError::SignatureExpired { key_id, expires_epoch }`.
5. Verifies the signature using the specified algorithm.
6. If any step fails for any directive, returns immediately with an error.

### 6.4 Changes to `sign_lockfile`

```rust
pub fn sign_lockfile(
    serialized_lockfile: &str,
    key_id: &str,
    algorithm: SignatureAlgorithm,
    private_key: &[u8],
    expires_epoch: u64,
) -> Result<String, SignatureError>
```

- `algorithm` is now explicit (defaults to Ed25519 in practice but must be specified).
- `expires_epoch` allows setting an expiration. Pass `0` for no expiration.
- `private_key` is a byte slice (32 bytes for Ed25519 seed, 57 bytes for Ed448).
- For Ed25519, the function accepts the 32-byte seed (expanded internally, matching v0.11 behavior). For Ed448, it accepts the 57-byte private key.

Multiple signatures are produced by calling `sign_lockfile` multiple times on the same base text. Each call appends a new `@signature` line.

### 6.5 Graph Query API (New Module: `graph`)

All graph functions operate on `&Lockfile` and use content IDs for identity (computed via `fnv::calculate`).

```rust
/// Returns package indices in topological order (dependencies before dependents).
///
/// Errors if a cycle is detected, returning the cycle as a Vec of package names.
pub fn topological_sort(lockfile: &Lockfile) -> Result<Vec<usize>, Vec<String>>

/// Returns the names of all packages that directly or transitively depend on `package_name`.
///
/// Returns an empty Vec if the package has no dependents.
pub fn dependents_of(lockfile: &Lockfile, package_name: &str) -> Vec<String>

/// Returns the names of all direct and transitive dependencies of `package_name`.
pub fn transitive_deps(lockfile: &Lockfile, package_name: &str) -> HashSet<String>

/// Returns packages that have no dependents (nothing depends on them).
/// These are candidates for removal when pruning unused packages.
pub fn leaf_packages(lockfile: &Lockfile) -> Vec<&Package>

/// Returns the names forming a dependency cycle, if one exists.
/// Returns None if the graph is acyclic.
pub fn detect_cycle(lockfile: &Lockfile) -> Option<Vec<String>>

/// Returns true if adding `new_deps` to `package_name` would create a cycle.
pub fn would_create_cycle(
    lockfile: &Lockfile,
    package_name: &str,
    new_deps: &[String],
) -> bool
```

**Implementation notes:**
- `topological_sort` uses Kahn's algorithm (BFS-based) for deterministic ordering among equally-ranked packages (lexicographic by name).
- `dependents_of` builds a reverse adjacency list and performs BFS from the target.
- `detect_cycle` uses DFS with a three-color marking scheme (white/gray/black).
- All functions are O(V + E) where V = number of packages and E = number of dependency edges.

### 6.6 New Error Variants

```rust
// In SignatureError:
#[error("Key '{key_id}' is not in the trusted key set")]
UntrustedKey { key_id: String },

#[error("Signature from '{key_id}' expired at epoch {expires_epoch}")]
SignatureExpired { key_id: String, expires_epoch: u64 },

#[error("Unsupported signature algorithm ID {algo_id}")]
UnsupportedSignatureAlgorithm { algo_id: u8 },

#[error("Ed448 verification failed")]
Ed448VerificationFailed,

// In Error (unchanged position, appended):
// (No new Error variants in this release — graph query errors use
//  Result<Vec<usize>, Vec<String>> for cycles and Ok for all other queries.)
```

## 7. Implementation Notes

### 7.1 BLAKE3 Trailer Computation

```rust
use blake3;

fn compute_trailer(payload_bytes: &[u8]) -> [u8; 32] {
    blake3::hash(payload_bytes).into_bytes()
}
```

The `blake3` crate is added as a dependency. During unpacking, the last 32 bytes are the expected digest. The verifier hashes bytes `[0..len-32]` and compares.

### 7.2 Ed448 Signature Support

Ed448 signatures require the `ed448-goldilocks` crate (or equivalent). The signature is 114 bytes, the public key is 57 bytes, and the private key is 57 bytes.

```rust
match algorithm {
    SignatureAlgorithm::Ed25519 => {
        // Existing ed25519-dalek logic
        assert_eq!(private_key.len(), 32);
        assert_eq!(signature_bytes.len(), 64);
    }
    SignatureAlgorithm::Ed448 => {
        // New Ed448 logic
        assert_eq!(private_key.len(), 57);
        assert_eq!(signature_bytes.len(), 114);
    }
}
```

### 7.3 Multi-Signature Semantics

The lockfile format does NOT encode a verification policy. The policy is entirely the verifier's responsibility:

- **Strict CI**: Provide a `trusted_keys` map with exactly the keys that MUST sign. If any `@signature` uses an unlisted key, `UntrustedKey` is returned.
- **Permissive local dev**: Provide an empty `trusted_keys` map. Unsigned lockfiles pass. Signed lockfiles also pass (the verifier finds no trusted keys and... wait, this is wrong).

Actually, let me clarify: the `trusted_keys` map has two valid modes:

1. **`trusted_keys` is empty AND no `@signature` directives exist** → `Ok(())` (unsigned, accepted).
2. **`trusted_keys` is empty AND `@signature` directives exist** → `Err(UntrustedKey)` (signed but no keys trusted = reject).
3. **`trusted_keys` is non-empty** → Every `@signature` MUST correspond to a trusted key and verify. If a `@signature` references an untrusted key, reject.

This ensures that a verifier that cares about signatures can enforce it, and a verifier that doesn't can still read unsigned lockfiles.

### 7.4 Topological Sort Determinism

When multiple packages have the same in-degree (no ordering constraint between them), they are sorted lexicographically by `name`. This ensures deterministic output across runs and platforms, which is essential for reproducible builds and diff-stable lockfiles.

### 7.5 Dependency Resolution Edge Cases

- **Self-dependency**: A package listing itself as a dependency is treated as a cycle of length 1. `detect_cycle` returns `["pkg"]`.
- **Missing dependency**: `dependents_of` and `transitive_deps` skip dependency names that don't match any package (same as `extract_subgraph` behavior — no error, just omission).
- **Duplicate dependencies**: If a package lists the same dependency twice (with different `DepType`), it appears as a single edge in the graph. The graph is built on name→name edges, not typed edges.

## 8. Migration Guide (v0.11 → v0.12)

### Payload Format

- **Version byte**: `0x07` → `0x08`
- **Trailer**: 4-byte CRC32 LE → 32-byte BLAKE3
- **Inner structs**: No changes

### Signature Format

- **v0.11**: `@signature <key_id> <sig>`
- **v0.12**: `@signature <key_id> 00 0 <sig>` (Ed25519, no expiration)

When migrating existing v0.11 signatures to v0.12, the signature *bytes* do NOT change (the signed message is the same text, the algorithm is the same). Only the directive syntax changes.

### Code Changes

```rust
// v0.11
let signed = sign_lockfile(&serialized, "ci@co.com", &expanded_key)?;
verify_signature(&signed, &public_key)?;

// v0.12
let signed = sign_lockfile(
    &serialized, "ci@co.com",
    SignatureAlgorithm::Ed25519,
    &seed,  // 32-byte seed, not 64-byte expanded key
    0,      // no expiration
)?;
let mut trusted = HashMap::new();
trusted.insert("ci@co.com".into(), (&public_key[..], SignatureAlgorithm::Ed25519));
verify_signature(&signed, &trusted)?;
```

Note the private key change: v0.12 `sign_lockfile` for Ed25519 accepts the 32-byte seed directly, not the 64-byte expanded key. This is cleaner and avoids the confusion of "which 64 bytes?" that existed in v0.11.

### CRC32 Removal

- `src/crc32.rs` module is deleted entirely.
- `crc32` is removed from `Cargo.toml` dependencies (it was never a dependency — CRC32 was implemented inline).
- `src/crc32.rs` file is deleted.
- Any code referencing `crate::crc32` is removed (only `validate_patches` used it, and it now uses BLAKE3).
- The `IntegrityCheckFailed` error variant is renamed to `PayloadDigestMismatch` for clarity, though the old name can be kept as an alias if desired.

### Graph API Adoption

Replace any ad-hoc graph traversal code with the new primitives:

```rust
// Before (manual):
let mut needed = HashSet::new();
// ... manual BFS ...

// After:
let needed = transitive_deps(&lockfile, "app");
```
