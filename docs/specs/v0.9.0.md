# HLOCK Specification v0.9.0

## 1. Overview
HLOCK v0.9.0 is **"The Platform & Provenance Release"**. It completes the peer dependency story by adding declarative peer requirements, introduces platform filter tags enabling a single universal lockfile to serve multi-platform installs, and adds Ed25519 lockfile signing — a feature no mainstream lockfile format offers.

### Design Principles
- **Single Lockfile, Many Platforms** — One `.hlock` file encodes all packages for all targets. Subgraph extraction accepts a platform filter to produce a pruned lockfile for a specific OS/arch combination.
- **Full Peer Lifecycle** — v0.8 recorded *how* a peer was satisfied. v0.9 adds *what* the peer requirement was, enabling installers to re-validate topology without re-resolving.
- **Tamper-Proof by Default** — An optional Ed25519 signature appended after the package block covers all preceding bytes. CI pipelines can verify the lockfile was produced by a trusted identity before installing.

**Backward Compatibility:** v0.9 is fully backward compatible with v0.8 lockfiles. The payload version byte remains `0x06`. New fields are appended to the end of the payload layout and are optional (zero-length counts). New header directives are appended after the package block and are ignored by v0.8 parsers.

---

## 2. File Structure

```text
<header directives>
<empty line>
<package lines>
<optional signature directive>
```

The only structural change from v0.8 is the optional addition of a signature directive after all package lines.

---

## 3. Header Directives

All v0.8 header directives are unchanged:

```text
@source <idx> <uri>
@override <name> <ver> -> <ver>
@feature <name> [flag1,flag2,...]
```

### 3.1 New: Signature Directive

```text
@signature <key_id> <base64url_signature>
```

| Field | Description |
|---|---|
| `key_id` | A human-readable identifier for the signing key (e.g., `ci@company.com`, `bot@github-actions`). Must not contain spaces. |
| `base64url_signature` | A 64-byte Ed25519 signature, Base64URL-encoded (no padding), covering every byte in the file from the first character through the newline *before* this `@signature` line. |

**Rules:**
- There MUST be at most one `@signature` directive per lockfile.
- It MUST appear after all package lines. No package lines may follow it.
- Verifiers MUST strip the `@signature` line (including its trailing newline) before computing the message to verify.
- If a verifier does not recognize the `@signature` directive (i.e., it is a v0.8 parser), it MUST ignore the line entirely without error.
- The Ed25519 verification context is the raw file bytes — no hashing, no framing, no prefix. The Ed25519 `verify(message, signature, public_key)` function is called directly.

**Signature Generation Pseudocode:**
```text
message = file_bytes[0 .. position_of_at_sign_line]
signature = ed25519_sign(message, private_key)
encoded = base64url_encode(signature)  // no padding
output += "@signature <key_id> <encoded>\n"
```

## 4. Content IDs

Unchanged from v0.8. Dependencies are still referenced by FNV-1a 64-bit hashes of `canonical_name@major.minor.patch`.

## 5. The Binary Payload (v0.9 Schema)

### 5.1 Payload Layout

The payload version byte remains `0x06`. v0.9 appends three new sections after the existing v0.8 `Dependencies` array and before the `CRC32` trailer.

| Field | Type | Size | Description |
|---|---|---|---|
| `Version` | Byte | 1 byte | `0x06` (unchanged) |
| `LogicalNameLen` | Varint | 1+ bytes | (unchanged) |
| `LogicalName` | Bytes | Variable | (unchanged) |
| `SourceIdx` | Varint | 1+ bytes | (unchanged) |
| `Major` | Varint | 1-9 bytes | (unchanged) |
| `Minor` | Varint | 1-9 bytes | (unchanged) |
| `Patch` | Varint | 1-9 bytes | (unchanged) |
| `HashCount` | Varint | 1+ bytes | (unchanged) |
| `Hashes` | Array | Variable | (unchanged) |
| `FeatureCount` | Varint | 1+ bytes | (unchanged) |
| `Features` | Array | Variable | (unchanged) |
| `PeerCount` | Varint | 1+ bytes | (unchanged) |
| `Peers` | Array | Variable | (unchanged) |
| `DepCount` | Varint | 1+ bytes | (unchanged) |
| `Dependencies` | Array | Variable | (unchanged) |
| `PeerReqCount` | Varint | 1+ bytes | **NEW:** Number of peer requirements. |
| `PeerRequirements` | Array | Variable | **NEW:** Array of peer requirement structs (see 5.2). |
| `PlatformTagCount` | Varint | 1+ bytes | **NEW:** Number of platform tags. |
| `PlatformTags` | Array | Variable | **NEW:** Array of platform tag structs (see 5.3). |
| `CRC32` | Bytes | 4 bytes LE | Checksum of all preceding bytes (unchanged scope). |

**v0.8 Compatibility:** A v0.8 parser reads through `Dependencies`, then expects `CRC32` at `cursor + 4 == bytes.len()`. A v0.9 payload with zero `PeerReqCount` and zero `PlatformTagCount` writes two `0x00` varint bytes before the CRC32, shifting it by 2 bytes. This means a v0.9-encoded payload with empty new sections is **NOT** byte-compatible with a strict v0.8 parser.

To maintain backward compatibility, serializers MUST detect whether any package uses new fields. If no package has peer requirements or platform tags, the serializer MUST emit a v0.8-compatible payload (omit `PeerReqCount`, `PlatformTagCount`, and their arrays entirely). If any package uses new fields, all packages in that lockfile MUST include the new sections (with zero counts for packages that don't use them).

**Implementation Note:** The `serialize` function accepts a `compat_mode: CompatMode` parameter:

```text
enum CompatMode {
    V8,   // Omit new sections entirely
    V9,   // Include new sections on every package
}
```

The `deserialize` function auto-detects: after reading `Dependencies`, if `cursor + 4 == bytes.len()`, parse as v0.8. Otherwise, parse the new sections.

### 5.2 Peer Requirement Struct Layout

Declares *what* a peer dependency requires, independent of how it was resolved.

| Field | Type | Size | Description |
|---|---|---|---|
| `PeerNameLen` | Varint | 1+ bytes | Length of the peer name string. |
| `PeerName` | Bytes | Variable | The name of the required peer (e.g., `react`). |
| `VersionRangeLen` | Varint | 1+ bytes | Length of the version range string. |
| `VersionRange` | Bytes | Variable | Semver range string (e.g., `^17.0.0`, `>=18.0.0`). May be empty (`""`) to indicate "any version" (common for peer deps without range constraints). |
| `IsOptional` | Byte | 1 byte | `0x00` (required) or `0x01` (optional peer). |

**Example:** `react-dom` requiring `react@">=17.0.0"`:
- `PeerName` = `react`
- `VersionRange` = `>=17.0.0`
- `IsOptional` = `0x00`

**Relationship to PeerResolution:** `PeerRequirements` and `PeerResolution` (from v0.8) are complementary. A package may have 3 peer requirements but only 2 peer resolutions if one peer is missing and was not required (`IsOptional = 0x01`). Installers MUST verify that every required peer (`IsOptional = 0x00`) has a corresponding entry in `resolved_peers` whose `satisfied_by_content_id` points to a package matching the version range.

### 5.3 Platform Tag Struct Layout

Declares that this package is only relevant on specific platforms. If the tag list is empty, the package is platform-agnostic (included on all platforms).

| Field | Type | Size | Description |
|---|---|---|---|
| `OS` | Byte | 1 byte | Target OS ID (see table below). |
| `Arch` | Byte | 1 byte | Target architecture ID (see table below). |

**OS IDs:**

| ID | OS |
|---|---|
| 0x00 | Any |
| 0x01 | Linux |
| 0x02 | MacOS |
| 0x03 | Windows |
| 0x04 | FreeBSD |
| 0x05 | Android |
| 0x06 | iOS |
| 0xFF | Unknown / Other |

**Architecture IDs:**

| ID | Arch |
|---|---|
| 0x00 | Any |
| 0x01 | X86_64 |
| 0x02 | Aarch64 |
| 0x03 | Wasm32 |
| 0x04 | Arm (32-bit) |
| 0x05 | S390x |
| 0x06 | Ppc64le |
| 0xFF | Unknown / Other |

**Semantics:** A package is included for a target platform if ANY of its platform tags match. A tag matches if both `OS` and `Arch` match the target, or either field is `0x00` (Any). If the tag list is empty, the package matches all platforms.

**Examples:**
- `[(Linux, X86_64)]` — only linux-x86_64
- `[(Linux, Any)]` — all linux architectures
- `[(Any, Aarch64)]` — aarch64 on any OS
- `[(Linux, X86_64), (MacOS, Aarch64)]` — linux-x86_64 OR macos-aarch64
- `[]` — all platforms (default)

## 6. Public API Specification

### 6.1 New Data Structures

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct PeerRequirement {
    pub peer_name: String,
    pub version_range: String,  // Empty string means "any version"
    pub is_optional: bool,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum TargetOS {
    Any,
    Linux,
    MacOS,
    Windows,
    FreeBSD,
    Android,
    IOS,
    Unknown,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub enum TargetArch {
    Any,
    X86_64,
    Aarch64,
    Wasm32,
    Arm,
    S390x,
    Ppc64le,
    Unknown,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct PlatformTag {
    pub os: TargetOS,
    pub arch: TargetArch,
}

pub enum CompatMode {
    V8,
    V9,
}
```

### 6.2 Changes to Existing Structures

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Package {
    pub name: String,
    pub logical_name: Option<String>,
    pub source_idx: usize,
    pub major: u64,
    pub minor: u64,
    pub patch: u64,
    pub hashes: Vec<IntegrityHash>,
    pub features: Vec<String>,
    pub resolved_peers: Vec<PeerResolution>,
    pub dependencies: Vec<Dependency>,
    pub peer_requirements: Vec<PeerRequirement>,   // NEW
    pub platform_tags: Vec<PlatformTag>,           // NEW
}
```

### 6.3 New Functions

```rust
/// Verify the Ed25519 signature on a lockfile string.
/// Returns Ok(()) if the signature is valid or absent.
/// Returns Err if the signature is present but invalid.
pub fn verify_signature(
    lockfile_content: &str,
    public_key: &[u8; 32],
) -> Result<(), SignatureError>;

/// Sign a serialized lockfile string with an Ed25519 private key.
/// Appends the @signature directive and returns the complete signed string.
pub fn sign_lockfile(
    serialized_lockfile: &str,
    key_id: &str,
    private_key: &[u8; 64],
) -> Result<String, SignatureError>;

/// Extract a subgraph filtered to a specific target platform.
/// Packages with non-empty platform_tags that don't match the
/// given (os, arch) are excluded, along with any dependencies
/// that become unreachable.
pub fn extract_subgraph_platform(
    lockfile: &Lockfile,
    root_content_ids: &[u64],
    target_os: TargetOS,
    target_arch: TargetArch,
) -> Result<Lockfile, Error>;

/// Serialize with explicit compat mode.
pub fn serialize_compat(lockfile: &mut Lockfile, mode: CompatMode) -> Result<String, Error>;
```

### 6.4 New Error Variants

```rust
#[derive(Debug, thiserror::Error)]
pub enum Error {
    // ... existing variants ...

    #[error("Invalid signature: {reason}")]
    InvalidSignature { reason: String },

    #[error("Required peer '{peer_name}' for package '{package}' has version range '{range}' but resolved version '{resolved}' does not satisfy it")]
    PeerRangeMismatch { package: String, peer_name: String, range: String, resolved: String },

    #[error("Required peer '{peer_name}' for package '{package}' has no resolution")]
    PeerRequirementUnsatisfied { package: String, peer_name: String },

    #[error("No packages match the target platform ({os:?}, {arch:?})")]
    NoPackagesForPlatform { os: TargetOS, arch: TargetArch },
}

#[derive(Debug, thiserror::Error)]
pub enum SignatureError {
    #[error("Base64URL decode failed: {0}")]
    InvalidBase64(#[from] &'static str),

    #[error("Ed25519 verification failed")]
    VerificationFailed,

    #[error("Malformed signature directive: {reason}")]
    MalformedDirective { reason: String },
}
```

### 6.5 Parser Behavior

- `deserialize`: After reading the `Dependencies` array, check if `cursor + 4 == bytes.len()`. If yes, parse CRC32 as v0.8. If no, parse `PeerReqCount`, `PeerRequirements`, `PlatformTagCount`, `PlatformTags`, then CRC32 as v0.9.
- `unpack_payload`: The `Version` byte remains `0x06` for both v0.8 and v0.9 payloads. The layout detection is structural, not version-flagged.
- `verify_signature`: Parse the last non-empty line. If it starts with `@signature`, extract `key_id` and signature, strip that line, and verify. If the last line is not `@signature`, return `Ok(())` (no signature to verify).
- `extract_subgraph_platform`: Identical to `extract_subgraph` but after computing the initial reachable set, filters out any package whose `platform_tags` list is non-empty and does not match `(target_os, target_arch)`, then re-computes reachability. Repeat until stable.

## 7. Implementation Notes for Package Managers

### 7.1 Platform-Aware Installation

A universal lockfile contains packages for all platforms. The installer workflow:

1. Detect host platform `(os, arch)`.
2. Call `extract_subgraph_platform(lockfile, &root_cids, os, arch)`.
3. Install only the resulting subgraph.

This eliminates the need for platform-specific lockfile files and prevents "works on my machine" bugs caused by committing the wrong platform's lockfile.

### 7.2 Peer Requirement Validation

After resolution and before writing the lockfile, the package manager SHOULD validate:

```text
for each package:
    for each peer_req where !is_optional:
        assert peer_req.peer_name in package.resolved_peers
        assert resolved_version satisfies peer_req.version_range
```

On install (reading the lockfile), the installer CAN re-validate this as a safety check. If validation fails, it MUST fail the install rather than silently proceeding with an invalid topology.

### 7.3 CI Signing Workflow

```text
# In CI, after lockfile generation:
hlock sign --key-id "ci@company.com" --key-file /secrets/hlock.key hlock.lock

# In CI, before install:
hlock verify --key-file /trusted-keys/ci.pub hlock.lock
```

If verification fails, the CI job MUST abort. The public key file should be committed to the repository or fetched from a trusted source (e.g., TUF metadata, GitHub deploy keys).

### 7.4 v0.8 Fallback for Mixed Ecosystems

If a team has some tooling that only understands v0.8, use `serialize_compat(&mut lockfile, CompatMode::V8)`. This strips all v0.9 fields. Packages with peer requirements or platform tags will lose that information. The serializer MUST emit a warning if any data is being discarded.

### 7.5 Platform Tag Best Practices

- **Native binaries** (e.g., `esbuild-darwin-arm64`, `@swc-win32-x64-msvc`): Single tag like `[(MacOS, Aarch64)]`.
- **OS-agnostic native** (e.g., `@napi-rs/cli` with per-OS binaries): Multiple tags like `[(Linux, Any), (MacOS, Any), (Windows, Any)]`.
- **Pure JS/TS packages**: Empty tag list `[]` (default, platform-agnostic).
- **Wasm packages**: `[(Any, Wasm32)]` — runs on any OS but only as Wasm.

Never use `[(Any, Any)]` explicitly; use an empty tag list instead (they are semantically identical, but empty is the canonical form).

## 8. Migration Guide (v0.8 → v0.9)

| Area | Action |
|---|---|
| Existing lockfiles | No migration needed. v0.9 deserializers read v0.8 payloads transparently. |
| Serializers | Default to `CompatMode::V9`. Add `--compat v8` CLI flag if needed. |
| Package structures | Initialize `peer_requirements: vec![]` and `platform_tags: vec![]` on all existing packages. |
| Subgraph extraction | Continue using `extract_subgraph`. Switch to `extract_subgraph_platform` when platform filtering is needed. |
| Signature verification | Optional. Add to CI pipelines incrementally. Lockfiles without signatures are still valid. |
