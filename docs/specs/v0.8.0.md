# HLOCK Specification v0.8.0

## 1. Overview
HLOCK v0.8.0 is **"The Monorepo Topology Release"**. It introduces First-Class Package Aliasing, Peer Dependency Topology Tracking, and Content-Addressable Storage (CAS) source URIs. 

**Important Notice:** HLOCK is currently in an active alpha iteration phase. To prevent protocol bloat and allow the architecture to evolve rapidly, **all backward compatibility with v0.05 and older payloads has been permanently dropped.** Parsers MUST strictly reject any payload version less than `0x06`.

By combining Aliasing, Peer Tracking, and the existing Sparse Subgraph Extraction API, HLOCK becomes the only lockfile format capable of flawlessly modeling complex enterprise monorepo topologies (e.g., running React v18 and v19 simultaneously in the same tree) while strictly enforcing SLSA provenance.

## 2. File Structure

```text
<header directives>
<empty line>
<package lines>
```

### 2.1 The Metadata Header
*   `@source <idx> <uri>`: Unchanged conceptually, but URI scheme expanded.
*   `@override <name> <ver> -> <ver>`: Unchanged.
*   `@feature <name> [flag1,flag2,...]`: Unchanged.

**New Source URI Schemes:**
```text
@source 0 https://registry.npmjs.org/
@source 1 cas+https://cas.my-company.com/
@source 2 ipfs://QmXyZ1...
```

## 3. Content IDs
Unchanged. Dependencies are still referenced by FNV-1a 64-bit hashes of `canonical_name@version`. This guarantees merge-safety even when logical names are aliased.

## 4. The Binary Payload (v0.8 Schema)

### 4.1 Payload Layout

| Field | Type | Size | Description |
| :--- | :--- | :--- | :--- |
| `Version` | Byte | 1 byte | **MUST be `0x06`**. Parsers MUST reject `0x05` and older. |
| `LogicalNameLen` | Varint | 1+ bytes | Length of the logical name string. |
| `LogicalName` | Bytes | Variable | The name code uses to import the package (e.g., `react-v18`). If `0`, the canonical name on the text line is used. |
| `SourceIdx`| Varint | 1+ bytes | Index mapping to `@source <idx>`. |
| `Major` | Varint | 1-9 bytes | Package major version (Canonical). |
| `Minor` | Varint | 1-9 bytes | Package minor version (Canonical). |
| `Patch` | Varint | 1-9 bytes | Package patch version (Canonical). |
| `HashCount`| Varint | 1+ bytes | Number of integrity hashes. |
| `Hashes` | Array | Variable| Array of extended hash structs (unchanged from v0.7). |
| `FeatureCount`| Varint | 1+ bytes | Number of strings in this package's local Feature Table. |
| `Features` | Array | Variable| Array of `(VarintLen, UTF-8 Bytes)`. |
| `PeerCount`| Varint | 1+ bytes | **NEW:** Number of resolved peer dependencies. |
| `Peers` | Array | Variable| **NEW:** Array of Peer Resolution structs (see 4.2). |
| `DepCount`| Varint | 1+ bytes | Number of direct dependencies. |
| `Dependencies`| Array | Variable| Array of dependency structs (unchanged from v0.5). |
| `CRC32` | Bytes | 4 bytes | Standard CRC32 (IEEE 802.3) checksum of all preceding bytes, little-endian. |

### 4.2 Peer Resolution Struct Layout
Explicitly records *how* a peer dependency was satisfied in the dependency tree, completely eliminating guesswork by the installer.

| Field | Type | Size | Description |
| :--- | :--- | :--- | :--- |
| `PeerNameLen` | Varint | 1+ bytes | Length of the peer name string. |
| `PeerName` | Bytes | Variable | The name of the required peer (e.g., `react`). |
| `SatisfiedByContentID`| Bytes | 8 bytes | FNV-1a 64-bit hash of the package that satisfies this peer. |
| `IsHoistedToRoot` | Byte | 1 byte | `0x00` (Nested) or `0x01` (Hoisted to workspace root). Hint for symlink structures. |

## 5. Public API Specification

### 5.1 Core Data Structures

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum Source {
    Registry(String),
    Local(String),
    Git(String),
    Workspace,
    CasHttp(String), // NEW
    Ipfs(String),     // NEW
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct PeerResolution {
    pub peer_name: String,
    pub satisfied_by_content_id: u64,
    pub is_hoisted_to_root: bool,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Package {
    pub name: String,               // The Canonical Name
    pub logical_name: Option<String>, // NEW: What code imports. None means == canonical.
    pub source_idx: usize,
    pub major: u64,
    pub minor: u64,
    pub patch: u64,
    pub hashes: Vec<IntegrityHash>,
    pub features: Vec<String>,
    pub resolved_peers: Vec<PeerResolution>, // NEW
    pub dependencies: Vec<Dependency>,
}
```

### 5.2 Parser Behavior
*   `unpack_payload`: If `Version < 0x06`, return `Error::UnknownPayloadVersion`. Do not attempt fallback logic.
*   If `LogicalNameLen == 0`, `logical_name` MUST be set to `None` in the `Package` struct.
*   `serialize`: Always write `LogicalNameLen` and `LogicalName`. If `logical_name` is `None`, write `0x00`.

## 6. Implementation Notes for Package Managers

### 6.1 Zero-Conflict Aliasing
To install two versions of the same package (e.g., migrating React v18 to v19):
1. The canonical package is `react@19.0.0`.
2. An alias package is added with canonical name `react@18.2.0` and `logical_name: Some("react-v18")`.
3. Dependencies that haven't migrated yet point their Content ID to `react@18.2.0`.
4. The installer reads `logical_name` and creates a symlink `node_modules/react-v18` pointing to the `react@18.2.0` cache folder. 

### 6.2 Strict Peer Resolution
When an installer encounters a `PeerResolution`, it MUST verify that the package located at `satisfied_by_content_id` is actually accessible to the current package based on the `is_hoisted_to_root` flag. If the topology is invalid, it MUST fail the install immediately rather than silently hoisting and causing runtime errors.

### 6.3 CAS Fetching
If a package's `Source` is `CasHttp`, the fetcher MUST construct a `GET` request to `{endpoint}/{sha256_of_digest}`. The `IntegrityHash` digest directly serves as the cache key, bypassing registry index lookups entirely.
