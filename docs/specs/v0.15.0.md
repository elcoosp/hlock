# HLOCK Specification v0.15.0

## 1. Overview

HLOCK v0.15.0 is **"The Audit, Trust & Interop Release"**. It addresses the three biggest gaps between hlock and production-grade lockfile systems: vulnerability awareness, cryptographic trust delegation, and ecosystem interoperability. No package manager in 2025 can be taken seriously without advisory scanning, and no lockfile format can claim supply-chain integrity without transparency logs and key rotation.

This is a non-breaking release from v0.14. The payload version byte remains `0x08`. The binary payload format is unchanged. All new functionality is additive.

### Design Principles

- **Advisories Are Data, Not Commands** — Advisory directives describe the state of the world at resolution time. They do not dictate install behavior. The consumer decides whether a vulnerability is acceptable.
- **Trust Is Delegated, Not Assumed** — A single signing key is a single point of failure. v0.15 introduces a trust root with key rotation and delegated authority, modeled on TUF (The Update Framework) and Sigstore.
- **Interop Is Import-Only** — hlock's binary format is strictly superior to JSON/YAML lockfiles. We import from other formats; we never export to them. Once data enters hlock, it stays in hlock.
- **Permission Is Deny-By-Default** — Lifecycle hooks (postinstall, etc.) are the #1 attack vector in npm. v0.15 makes hook execution opt-in with explicit allowlists.

## 3. New Header Directives

### 3.1 `@advisory`

```
@advisory <package> <advisory_id> <severity> <url> <affected_versions>
```

Records a known vulnerability affecting a package in the lockfile.

| Field | Format | Description |
|---|---|---|
| `package` | String | Package name (must match a package line) |
| `advisory_id` | String | Advisory identifier (e.g., `GHSA-xxxx-xxxx-xxxx`, `CVE-2024-12345`, `OSV-2024-567`) |
| `severity` | Enum string | One of: `critical`, `high`, `medium`, `low`, `info` |
| `url` | String | URL to the advisory details (may be empty) |
| `affected_versions` | String | Semver range of affected versions (e.g., `<4.17.21`) |

**Rules:**
- Multiple `@advisory` directives may reference the same package (different vulnerabilities).
- Advisory directives referencing packages not in the lockfile are retained (the package may be added later).
- `severity` must be one of the five enum values; unknown values are a parse error.
- `url` may be empty (represented as a zero-length string between spaces).
- `affected_versions` is informational; it does not affect install behavior.

**Example:**
```
@advisory lodash GHSA-jf85-cq4p-4qr8 high https://github.com/advisories/GHSA-jf85-cq4p-4qr8 <4.17.21
@advisory express CVE-2024-43999 medium  >=4.0.0 <4.21.0
```

### 3.2 `@license`

```
@license <package> <expression>
```

Records the SPDX license expression for a package.

| Field | Format | Description |
|---|---|---|
| `package` | String | Package name |
| `expression` | String | SPDX license expression (e.g., `MIT`, `Apache-2.0`, `(MIT OR Apache-2.0)`, `GPL-3.0-only WITH Classpath-exception-2.0`) |

**Rules:**
- At most one `@license` per package. Last-wins on duplicates.
- The expression is not validated at parse time (SPDX expression parsing is a separate concern).
- License data is used by the lint engine and SBOM generator.

**Example:**
```
@license lodash MIT
@license webpack MIT
@license core-js ISC
```

### 3.3 `@policy`

```
@policy <policy_type> <package_pattern> <value>
```

Declares an installation policy constraint.

| Field | Format | Description |
|---|---|---|
| `policy_type` | String | One of: `allow-hook`, `deny-hook`, `allow-script`, `deny-script`, `build-env`, `engine` |
| `package_pattern` | String | Package name, `*` for all, or glob pattern (`@scope/*`) |
| `value` | String | Policy-specific value (hook name, engine constraint, etc.) |

**Policy Types:**

| Type | Value Meaning | Example |
|---|---|---|
| `allow-hook` | Hook name that is permitted to execute | `allow-hook * postinstall` |
| `deny-hook` | Hook name that is forbidden | `deny-hook malicious-pkg postinstall` |
| `allow-script` | Script identifier permitted | `allow-hook * preinstall` |
| `deny-script` | Script identifier forbidden | `deny-script evil-pkg prepublish` |
| `build-env` | Pinned build environment | `build-env * node>=20.0.0` |
| `engine` | Required engine constraint | `engine app node>=22.0.0` |

**Rules:**
- `deny` takes precedence over `allow` when both match.
- `*` matches all packages.
- Glob patterns use `*` as the only wildcard (no `?`, no `[...]`).
- `build-env` and `engine` values use npm-compatible engine strings.
- Multiple `@policy` directives are allowed; they are evaluated in order (deny-wins).

**Example:**
```
@policy deny-hook * postinstall
@policy allow-hook lodash postinstall
@policy build-env * node>=20.11.0
@policy engine app node>=22.0.0
```

### 3.4 `@trust-root`

```
@trust-root <key_id> <algorithm> <public_key_b64url> <expires_epoch> <role>
```

Declares a trusted root key for the lockfile's trust hierarchy.

| Field | Format | Description |
|---|---|---|
| `key_id` | String | Key identifier (email, URL, or hash) |
| `algorithm` | Hex u8 | `00` = Ed25519, `02` = ML-DSA-65 |
| `public_key_b64url` | Base64URL | Encoded public key bytes |
| `expires_epoch` | Decimal u64 | Key expiration (0 = never) |
| `role` | String | One of: `root`, `targets`, `snapshot`, `delegation` |

**Rules:**
- At least one `@trust-root` with `role=root` must exist for the trust chain to be valid.
- `@trust-root` directives appear in the header, before the empty-line separator.
- Multiple root keys may exist (threshold signing).
- Key rotation: add a new root key with a later expiry, remove the old one in a subsequent commit.

**Example:**
```
@trust-root ci@company.com 00 dGhpc19pc19hX3B1YmxpY19rZXk 1735689600 root
@trust-root bot@github-actions 02 QW5vdGhlclB1YmxpY0tleQ 1735689600 targets
```

### 3.5 `@mirror`

```
@mirror <scope> <url>
```

Maps a package scope to a specific registry mirror.

| Field | Format | Description |
|---|---|---|
| `scope` | String | Package scope (e.g., `@internal`, `@company`, or `*` for default) |
| `url` | String | Registry URL |

**Example:**
```
@mirror @internal https://npm.company.com/
@mirror @acme https://acme.jfrog.io/artifactory/npm/
@mirror * https://registry.npmmirror.org/
```

### 3.6 `@compat`

```
@compat <min_version>
```

Declares the minimum hlock version required to safely interpret this lockfile.

| Field | Format | Description |
|---|---|---|
| `min_version` | Semver string | Minimum version (e.g., `0.14.0`) |

**Rules:**
- `@compat` is optional. If absent, any version may attempt to read.
- A reader encountering a `@compat` version higher than its own should emit a warning.
- This is a hint, not a hard gate. Format-breaking changes would use a new payload version byte.

**Example:**
```
@compat 0.14.0
```

### 5.6 Format Import

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ImportFormat {
    NpmLockJson,     // package-lock.json v2/v3
    YarnLock,        // yarn.lock (v1 berry and classic)
    PnpmLockYaml,    // pnpm-lock.yaml
}

pub fn import_lockfile(
    content: &str,
    format: ImportFormat,
) -> Result<Lockfile, Error>
```

**Implementation notes:**
- `import_lockfile` parses the foreign format and produces a `Lockfile` with:
  - `Source::Registry` entries derived from `resolved` URLs
  - Dependencies mapped to `DepType::Runtime` (npm doesn't distinguish dev/properly in the lockfile body; `dev` is inferred from location)
  - `IntegrityHash` from `integrity` fields (converted from `sha512-xxx` base64 format)
  - `@license` directives populated from `license` fields if present
  - `packages[].name` set from the package name
  - `workspace_pkgs` populated from `workspaces` fields
- The resulting `Lockfile` can be serialized via `serialize()` to produce a valid `.hlock` file.
- Import is lossy: features that have no hlock equivalent are dropped silently.

## 11. Import Format Specifications

### 11.1 `package-lock.json` (v2/v3)

```rust
fn import_npm_lockjson(content: &str) -> Result<Lockfile, Error>
```

**Mapping:**

| package-lock.json | hlock |
|---|---|
| `name` | `workspace_root` or first `WorkspacePkg` |
| `lockfileVersion` | (ignored — format is inferred) |
| `packages[""].dependencies` | top-level `Package.dependencies` (Runtime) |
| `packages[""].devDependencies` | top-level `Package.dependencies` (Dev) |
| `packages["node_modules/foo"].version` | `major.minor.patch` |
| `packages["node_modules/foo"].resolved` | `Source::Registry(url)` or `Source::Git(url)` |
| `packages["node_modules/foo"].integrity` | `IntegrityHash` (parse `sha512-XXXXX` base64) |
| `packages["node_modules/foo"].license` | `@license` directive |
| `packages["node_modules/foo"].inBundle` | (dropped) |
| `packages["node_modules/foo"].link` | (dropped — no hlock equivalent) |

**Integrity conversion:** npm uses `<algo>-<base64>` format. Convert:
- `sha512-abc123` → `IntegrityHash { algo: Sha512, digest: base64_decode("abc123") }`
- `sha1-abc123` → `IntegrityHash { algo: Sha1, digest: base64_decode("abc123") }`
- `sha256-abc123` → `IntegrityHash { algo: Sha256, digest: base64_decode("abc123") }`

Note: npm uses standard base64 (not base64url) for integrity strings. The import function must handle this difference.

### 11.2 `yarn.lock` (v1 classic)

```rust
fn import_yarn_lock(content: &str) -> Result<Lockfile, Error>
```

**Mapping:**

| yarn.lock | hlock |
|---|---|
| `"foo@^1.0.0"` | `@provenance foo ^1.0.0 (parent) 0 0 N` |
| `version "1.2.3"` | `major.minor.patch` |
| `resolved "https://...#hash"` | `Source::Registry(url)`, integrity from `#hash` |
| `integrity "sha512-..."` | `IntegrityHash` |

### 11.3 `pnpm-lock.yaml`

```rust
fn import_pnpm_lock(content: &str) -> Result<Lockfile, Error>
```

**Mapping:**

| pnpm-lock.yaml | hlock |
|---|---|
| `importers["."].dependencies` | Runtime deps |
| `importers["."].devDependencies` | Dev deps |
| `packages."/foo/1.0.0"` | Package entry |
| `packages."/foo/1.0.0".resolution` | Source + integrity |

## 12. File Locking

### 12.1 `.hlock.lock` File

When a process opens a lockfile for writing, it MUST create a companion `.hlock.lock` file containing:

```
<pid>
<hostname>
<acquired_epoch>
```

**API:**

```rust
pub struct LockfileGuard {
    path: std::path::PathBuf,
    lock_path: std::path::PathBuf,
}

impl LockfileGuard {
    /// Acquire an exclusive lock on the lockfile.
    /// Blocks until the lock is available or timeout expires.
    pub fn acquire(
        path: &std::path::Path,
        timeout: std::time::Duration,
    ) -> Result<LockfileGuard, Error>

    /// Read the lock metadata (who holds it, when acquired).
    pub fn lock_info(&self) -> Result<Option<LockInfo>, Error>

    /// Release the lock (dropped automatically on Drop).
    pub fn release(self)
}

impl Drop for LockfileGuard {
    fn drop(&mut self) {
        let _ = std::fs::remove_file(&self.lock_path);
    }
}

#[derive(Debug, Clone)]
pub struct LockInfo {
    pub pid: u32,
    pub hostname: String,
    pub acquired_epoch: u64,
}
```

**Implementation notes:**
- Uses `flock(2)` on Unix, `LockFileEx` on Windows (via a simple cross-platform abstraction).
- The `.hlock.lock` file is advisory; stale locks (process no longer running) are auto-cleaned.
- This is a runtime concern, not a format concern. The `.hlock.lock` file is never committed.

## 14. Version Compatibility

| Aspect | v0.14 | v0.15 |
|---|---|---|
| Payload version byte | `0x08` | `0x08` (unchanged) |
| Binary payload layout | Unchanged | Unchanged |
| `@advisory` directive | Not recognized | Parsed on deserialize |
| `@license` directive | Not recognized | Parsed on deserialize |
| `@policy` directive | Not recognized | Parsed on deserialize |
| `@trust-root` directive | Not recognized | Parsed on deserialize |
| `@mirror` directive | Not recognized | Parsed on deserialize |
| `@compat` directive | Not recognized | Parsed on deserialize |
| v0.14 lockfile by v0.15 | ✅ | New fields default to empty |
| v0.15 lockfile by v0.14 | ✅ | New `@`-directives skipped (v0.14's parser skips unknown `@`-prefixed lines in the package section; header directives would cause errors) |

**Note:** v0.14 cannot read v0.15 files that contain `@mirror`, `@trust-root`, `@policy`, or `@compat` in the header section. These are header directives that v0.14's parser doesn't recognize. The `@compat` directive exists specifically to allow tooling to detect this mismatch gracefully.
