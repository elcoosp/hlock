# HLOCK Specification v0.6.0

## 1. Overview
HLOCK v0.6.0 is a zero-schema-bump release that introduces high-level Graph Manipulation APIs.

While v0.5.0 established the merge-safe, content-addressable binary foundation, v0.6.0 leverages that architecture to expose native support for monorepo workflows. By operating entirely on the sorted package array and 64-bit Content IDs, v0.6.0 introduces algorithms for Semantic Diffing (incremental fetches) and Sparse Subgraph Extraction (zero-dependency CI) that run in O(N) time without decoding a single binary payload.

Schema Impact: None. The binary payload version remains 0x04. The text format remains unchanged.

## 2. File Structure
Unchanged from v0.5.0.

## 3. The Binary Payload (v0.5 Schema)
Unchanged. All v0.5.0 parsers will seamlessly support v0.6.0 files, as the on-disk format is identical.

## 4. Public API Specification

### 4.1 Semantic Differ
Calculates the exact set of changes between two lockfiles. Because packages are sorted alphabetically by serialize(), this can be computed via a fast two-pointer array merge rather than a naive O(N^2) lookup.

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub enum PackageChange {
    Added(Package),
    Removed(Package),
    Altered(Package, Package), // (old, new)
}

#[derive(Debug, Clone)]
pub struct LockfileDiff {
    pub changes: Vec<PackageChange>,
    pub unchanged_count: usize,
}

/// Computes the semantic diff between two lockfiles.
/// Note: This operates purely on the high-level Package structs.
/// To diff lockfiles without deserializing payloads (e.g., raw text diff),
/// implementations should compare raw Base64 strings after splitting by \t.
pub fn diff_lockfiles(old: &Lockfile, new: &Lockfile) -> LockfileDiff
```

Algorithm:
1. Assume old.packages and new.packages are sorted alphabetically (as guaranteed by serialize).
2. Use two pointers (i for old, j for new).
3. If old[i].name == new[j].name: Compare version (major.minor.patch) and hashes. If identical, increment unchanged_count. If different, push Altered(old[i], new[j]). Increment both i and j.
4. If old[i].name < new[j].name: Push Removed(old[i]). Increment i.
5. If old[i].name > new[j].name: Push Added(new[j]). Increment j.

### 4.2 Sparse Subgraph Extractor
Extracts a fully valid, standalone Hlock file containing only the transitive closure of a specified set of root packages. This is the key to Zero-Dependency CI in monorepos.

```rust
pub fn extract_subgraph(
    lockfile: &Lockfile,
    root_content_ids: &[u64],
) -> Result<Lockfile, Error>
```

Algorithm:
1. Initialize a HashSet<u64> called allowed_ids with the contents of root_content_ids.
2. Validate that all root_content_ids exist in lockfile.packages. If not, return Error::RootContentIdMissing.
3. Initialize a Vec<Package> for the output.
4. Graph Closure: Iterate through lockfile.packages. For each package, compute its Content ID. If the ID is in allowed_ids:
   - Add the package to the output.
   - Compute the Content IDs for all of its dependencies (using dep_name@dep_version lookup in the original lockfile) and insert them into allowed_ids.
5. Repeat: Because new IDs might have been added in step 4, re-scan lockfile.packages until a full pass yields no new packages. (In practice, a single reversed pass or a queue-based BFS is sufficient, but a retry loop is safest against edge cases).
6. Source Pruning: Build a new sources array. Iterate through the extracted packages, collect all unique source_idx values, and map them to a contiguous 0..N range. Update the source_idx on the extracted packages accordingly. Discard unused sources.
7. Override Pruning: Keep all @override directives (they apply globally regardless of whether the target is in the subgraph, to prevent accidental resolution to the wrong version if the subgraph is later merged).
8. Feature Pruning: Keep all @feature directives.
9. Return the newly constructed Lockfile.

## 5. Implementation Notes for Package Managers
- Incremental Installs: When a user runs install and the lockfile changes, package managers should use diff_lockfiles to fetch only Added and Altered packages, and delete Removed packages from the cache. This skips hashing/verifying the unchanged_count packages entirely.
- Monorepo CI: When running CI for a specific workspace package (e.g., apps/web), the package manager should compute the Content ID of apps/web, call extract_subgraph, write the result to a temporary file, and pass that to the package fetcher. This guarantees CI only downloads what apps/web actually needs, ignoring unrelated apps in the monorepo.
