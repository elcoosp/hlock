# HLOCK Specification v0.14.0

## 1. Overview

HLOCK v0.14.0 is **"The Supply Chain & Merge Intelligence Release"**. It solves the two most painful operational problems with lockfiles at scale: merge conflicts in monorepo branching workflows, and O(n) full-parse costs that make 100k+ package lockfiles impractical to query. It also brings compliance-ready SBOM generation, a declarative linting engine, and resolution provenance tracking.

This is a non-breaking release from v0.13. The payload version byte remains `0x08`. The binary payload format is unchanged. All new functionality is additive — new functions, new header directives, new types.

### Design Principles

- **Merge, Don't Re-resolve** — A lockfile merge must never invoke a resolver. It is a deterministic, structural 3-way merge over sorted package lists. When the merge algorithm cannot reconcile a conflict, it reports it precisely. The human decides.
- **Pay For What You Use** — A CI step that checks one package's hash should not parse 99,999 others. The lazy parser indexes line offsets in a single linear scan (no payload decoding), then decodes individual payloads on demand via binary search.
- **Compliance Without Scripts** — SBOM generation, lint rules, and provenance tracking are first-class library features, not post-processing scripts that drift out of sync with the format.
- **Governance as Code** — Lint rules are trait objects. Teams compose rule sets, share them across repos, and enforce them in CI. The library ships with a standard rule set covering the most common governance requirements.

## 3. Header Directives

### 3.1 New Directive: `@provenance`

```
@provenance <package> <constraint> <constrained_by> <dep_type_id> <source_type_id> <depth>
```

Records the resolution provenance for a package — why this specific version is in the lockfile.

| Field | Format | Description |
|---|---|---|
| `package` | String | Package name (must match a package line) |
| `constraint` | String | Semver constraint that was satisfied (e.g. `^4.17.0`) |
| `constrained_by` | String | Name of the package that imposed this constraint (empty string for direct dependencies) |
| `dep_type_id` | Decimal | Same IDs as payload: `0` = Runtime, `1` = Dev, `2` = Peer, `3` = Optional, `4` = OptionalTarget |
| `source_type_id` | Decimal | `0` = Registry, `1` = Local, `2` = Git, `3` = Workspace, `4` = CasHttp, `5` = Ipfs |
| `depth` | Decimal | Dependency depth: `0` = direct, `1` = dependency of direct, etc. |

**Rules:**
- At most one `@provenance` directive per package name. If duplicates exist, the last one wins.
- A `@provenance` directive referencing a package name not present in the lockfile is ignored by parsers but should produce a lint warning.
- `depth` is informational. It does not affect resolution or installation.

**Example:**
```
@provenance lodash ^4.17.0 app 0 0 2
@provenance react ^18.0.0 app 0 0 1
@provenance jest ^29.0.0 app 1 0 1
@provenance pretty-format ^29.0.0 jest 0 0 2
```

### 3.2 Unchanged Directives

All v0.13 directives are unchanged: `@source`, `@override`, `@feature`, `@workspace-root`, `@workspace-pkg`, `@hoist-boundary`, `@metadata`, `@artifact`, `@patch`, `@digest`, `@signature`.

## 6. Public API Specification

### 6.1 Three-Way Lockfile Merge

```rust
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct MergeConflict {
    pub package_name: String,
    pub base: Option<Package>,
    pub ours: Package,
    pub theirs: Package,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum ConflictStrategy {
    Ours,
    Theirs,
    Fail,
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct MergeResult {
    pub lockfile: Lockfile,
    pub conflicts: Vec<MergeConflict>,
}

pub fn merge_lockfiles(
    base: &Lockfile,
    ours: &Lockfile,
    theirs: &Lockfile,
    strategy: ConflictStrategy,
) -> Result<MergeResult, Error>
```

**Merge algorithm:**

1. **Sources**: Merge source lists. If both branches added a source at the same index with different URLs, that is a conflict. New sources from either branch are appended with renumbered indices.
2. **Packages**: Three-way merge over the sorted package list:
   - Present in base, unchanged in both → keep
   - Present in base, altered in ours only → take ours
   - Present in base, altered in theirs only → take theirs
   - Present in base, altered in both → if identical, keep; otherwise conflict
   - Absent in base, added in ours only → include, remap source index
   - Absent in base, added in theirs only → include, remap source index
   - Absent in base, added in both → if identical, keep; otherwise conflict
   - Present in base, removed in ours → remove
   - Present in base, removed in theirs → remove
   - Present in base, removed in one and altered in other → conflict
3. **Overrides**: Take union. If both branches override the same package, take the more restrictive version (or conflict if different).
4. **Features**: Take union.
5. **Metadata**: Take union.
6. **Artifacts/Patches**: Take union.
7. **Provenance**: Re-derive after merge (or take union and remove entries for removed packages).
8. **Hoist boundaries**: Take union.

**Conflict resolution** depends on `strategy`:
- `Fail`: Return an `Error` containing all conflicts.
- `Ours`: Take `ours` version for every conflict.
- `Theirs`: Take `theirs` version for every conflict.

After merge, the result is re-serialized to ensure consistent source index mapping and sorted package order.

### 6.2 Lazy / Zero-Copy Parsing

```rust
#[derive(Debug, Clone)]
pub struct LazyLockfile {
    /* private fields */
}

#[derive(Debug, Clone, PartialEq, Eq)]
pub struct LockfileHeader {
    pub sources: Vec<Source>,
    pub overrides: Vec<Override>,
    pub features: Vec<(String, Vec<String>)>,
    pub metadata: Vec<(String, String)>,
    pub workspace_root: Option<String>,
    pub workspace_pkgs: Vec<WorkspacePkg>,
    pub hoist_boundaries: Vec<HoistBoundary>,
    pub artifacts: Vec<ArtifactDirective>,
    pub patches: Vec<PatchDirective>,
}

impl LazyLockfile {
    /// Scan the lockfile content, parsing only the header and indexing
    /// package line offsets. No binary payloads are decoded.
    pub fn scan(content: &str) -> Result<LazyLockfile, Error>

    /// Number of packages in the lockfile (O(1)).
    pub fn package_count(&self) -> usize

    /// Iterator over package names in sorted order (O(1) per iteration).
    pub fn package_names(&self) -> impl Iterator<Item = &str>

    /// Parse and return a single package by name (O(log n) + single payload decode).
    pub fn get_package(&self, name: &str) -> Result<Option<Package>, Error>

    /// Parse and return all packages from a given source index.
    pub fn get_packages_by_source(&self, source_idx: usize) -> Result<Vec<Package>, Error>

    /// Parse and return all packages whose names match a predicate.
    pub fn get_packages_where(
        &self,
        predicate: impl Fn(&str) -> bool,
    ) -> Result<Vec<Package>, Error>

    /// Access the parsed header (sources, overrides, features, etc.).
    pub fn header(&self) -> &LockfileHeader

    /// Validate the @digest directive if present (without parsing payloads).
    pub fn validate_digest(&self) -> Result<(), Error>

    /// Convert to a full Lockfile by parsing all remaining payloads.
    pub fn into_full(self) -> Result<Lockfile, Error>
}
```

**Implementation constraints:**
- `scan()` must be O(n) in lines but must not decode any Base64URL payload data.
- Package index entries store: `(name: String, line_start: usize, line_end: usize)`.
- `get_package()` binary-searches the sorted index, then calls `unpack_payload()` on the single relevant line.
- `LazyLockfile` holds an `Arc<String>` (or `Arc<str>`) of the raw content for zero-copy name borrowing.

### 6.6 New Error Variants

The following variants are added to the existing `Error` enum in `error.rs`:

```rust
#[derive(Debug, thiserror::Error)]
pub enum Error {
    // ... existing variants ...

    #[error("Merge conflict on package '{package_name}': {description}")]
    MergeConflict {
        package_name: String,
        description: String,
    },

    #[error("Merge conflict on source index {index}: ours has '{ours_url}', theirs has '{theirs_url}'")]
    MergeSourceConflict {
        index: usize,
        ours_url: String,
        theirs_url: String,
    },

    #[error("Merge failed with {count} conflict(s)")]
    MergeFailed { count: usize },

    #[error("Package '{package}' was removed in one branch but altered in another")]
    MergeRemoveAlterConflict { package: String },

    #[error("Override conflict on '{name}': ours has '{ours}', theirs has '{theirs}'")]
    MergeOverrideConflict {
        name: String,
        ours: String,
        theirs: String,
    },

    #[error("Lazy lockfile index corrupt at line {line_number}: {reason}")]
    LazyIndexCorrupt { line_number: usize, reason: String },

    #[error("SBOM generation failed for package '{package}': {reason}")]
    SbomGenerationFailed { package: String, reason: String },

    #[error("SBOM generation requires at least one registry source")]
    SbomNoRegistrySource,

    #[error("Provenance directive references unknown dep type {type_id}")]
    UnknownProvenanceDepType { type_id: u8 },

    #[error("Provenance directive references unknown source type {type_id}")]
    UnknownProvenanceSourceType { type_id: u8 },

    #[error("Duplicate @provenance directive for package '{package}'")]
    DuplicateProvenance { package: String },
}
```

**Design note:** When `ConflictStrategy::Fail` is used, `merge_lockfiles` returns `Err(Error::MergeFailed { count })` and the individual `MergeConflict` structs are not accessible. Callers who want to inspect conflicts before deciding must use `ConflictStrategy::Ours` or `ConflictStrategy::Theirs` and examine the `conflicts` field of the returned `MergeResult`. This two-path design avoids the anti-pattern of encoding recoverable data inside an `Error`.

### 6.9 Lazy Lockfile Implementation Specification

#### 6.9.1 Index Structure

`LazyLockfile` stores:

```rust
pub struct LazyLockfile {
    content: Arc<str>,
    header: LockfileHeader,
    index: Vec<IndexEntry>,
    provenance: Vec<ResolutionProvenance>,
}

#[derive(Debug, Clone)]
struct IndexEntry {
    name: String,        // package name, for binary search
    line_start: usize,   // byte offset of the line start in content
    line_end: usize,     // byte offset of the line end (\n or EOF)
}
```

#### 6.9.2 `scan()` Algorithm

1. Parse the header (same logic as `parse_header`), advancing through lines until the empty-line separator.
2. After the separator, iterate remaining lines:
   - Skip empty lines.
   - Skip `@artifact`, `@patch`, `@provenance`, `@digest`, `@signature` lines (but parse `@provenance` and `@artifact`/`@patch` into the header struct).
   - For lines containing `\t`: extract the package name (bytes before the first tab), record `(name, line_start, line_end)`.
3. The index is sorted by `name` (lockfiles are already serialized in sorted order, but `scan()` must verify this and sort if necessary, returning `LazyIndexCorrupt` if duplicate names are found).
4. Header-level `@provenance` directives are parsed and stored in the `provenance` field.

**Complexity:** O(n) in lines, O(1) per package entry (no Base64URL decode, no `unpack_payload`).

#### 6.9.3 `get_package()` Algorithm

1. Binary search `self.index` by `name`.
2. If not found, return `Ok(None)`.
3. Extract the line: `&self.content[line_start..line_end]`.
4. Split at `\t` to get the Base64URL payload portion.
5. Call `base64url::decode` then `unpack_payload` on the result.
6. Construct and return `Ok(Some(package))` using the same dependency-resolution logic as `deserialize` (requires the content-ID-to-name mapping built during `scan`).

**Complexity:** O(log n) for the search, O(k) for decoding a single payload of size k.

#### 6.9.4 `validate_digest()` Implementation

1. Find the `@digest` line in `content`.
2. If none, return `Ok(())`.
3. Compute `blake3::hash` over everything before the `@digest` line (same boundary logic as `whole_lockfile_digest`).
4. Compare. Return `Err(Error::DigestMismatch { .. })` on mismatch.

This does not require decoding any payload.

#### 6.9.5 `into_full()` Implementation

1. Iterate all index entries.
2. Decode each one via the same logic as `get_package()`.
3. Construct a `Lockfile` from the header, decoded packages, provenance, artifacts, and patches.

### 6.10 Merge Algorithm Specification

#### 6.10.1 Source Merging

```
fn merge_sources(base, ours, theirs) -> Result<(Vec<Source>, SourceIndexMap), Error>
```

1. Start with `base.sources` as the common foundation.
2. Walk `ours.sources` and `theirs.sources` in parallel:
   - If both branches agree on a source at index `i`, keep it.
   - If one branch added a source at a new index, append it.
   - If both branches added a different source at the same index, emit `MergeSourceConflict`.
3. Build a `SourceIndexMap`: `HashMap<(Branch, OldIdx), NewIdx>` so packages can be remapped.

#### 6.10.2 Package Merging

1. Convert each lockfile's package list into a `BTreeMap<String, Package>` keyed by name.
2. Compute three sets: `base_keys`, `ours_keys`, `theirs_keys`.
3. For each package name in the union of all keys:

| In base? | In ours? | In theirs? | Action |
|---|---|---|---|
| ✅ | ✅ same | ✅ same | Keep base |
| ✅ | ✅ altered | ✅ same | Take ours |
| ✅ | ✅ same | ✅ altered | Take theirs |
| ✅ | ✅ altered | ✅ altered | If identical, keep. Else conflict. |
| ✅ | ❌ | ✅ | Removed in ours. If theirs == base, remove. If theirs altered, `MergeRemoveAlterConflict`. |
| ✅ | ✅ | ❌ | Removed in theirs. If ours == base, remove. If ours altered, `MergeRemoveAlterConflict`. |
| ✅ | ❌ | ❌ | Remove (both deleted) |
| ❌ | ✅ | ❌ | Add ours (remap source index) |
| ❌ | ❌ | ✅ | Add theirs (remap source index) |
| ❌ | ✅ | ✅ | If identical, keep. Else conflict. |
| ❌ | ❌ | ❌ | Impossible (not in union) |

4. "Identical" comparison uses `Package` `PartialEq` (all fields including hashes, dependencies, features, etc.).
5. Source index remapping: when including a package from `ours` or `theirs`, translate its `source_idx` through the `SourceIndexMap`.

#### 6.10.3 Override Merging

1. Key overrides by `(name, from_version)`.
2. If both branches override the same key to the same `to_version`, keep.
3. If different `to_version`, emit `MergeOverrideConflict`.
4. Override-only additions from either branch are included.

#### 6.10.4 Remaining Fields

| Field | Strategy |
|---|---|
| `features` | Union by key. If same key has different values, take the longer list. |
| `metadata` | Union by key. If same key has different values, take `ours`. |
| `workspace_root` | If both present and different, take `ours`. |
| `workspace_pkgs` | Union by name. If same name has different path, take `ours`. |
| `hoist_boundaries` | Union by cosine name. If same name has different allowed_deps, take the intersection (more restrictive). |
| `artifacts` | Union by `(content_id, os_id, arch_id)`. Conflict if same key has different path. |
| `patches` | Union by `(content_id, patch_type)`. Conflict if same key has different path. |
| `provenance` | Union by package name. If both branches have provenance for the same package, take `ours`. Remove entries for packages not in the merged result. |

#### 6.10.5 Post-Merge

1. Sort packages by name.
2. Re-serialize the merged lockfile to ensure consistency.
3. Return `MergeResult { lockfile, conflicts }` where `conflicts` contains all `MergeConflict` entries encountered (empty if no conflicts).

### 6.11 SBOM Generation Specification

#### 6.11.1 PURL Construction

Package URLs follow the [purl spec](https://github.com/package-url/purl-spec):

```
pkg:npm/<name>@<version>
```

For scoped packages:
```
pkg:npm/%40scope/name@version
```

The namespace is always `npm` in this version. Future versions may support other ecosystems via a `@purl-namespace` metadata directive.

#### 6.11.2 Hash Algorithm Mapping

| HLOCK `HashAlgorithm` | SPDX Algorithm Identifier | CycloneDX Algorithm |
|---|---|---|
| `Sha1` | `SHA1` | `SHA-1` |
| `Sha256` | `SHA256` | `SHA-256` |
| `Sha512` | `SHA512` | `SHA-512` |
| `Blake3` | `blake3` (custom) | `BLAKE3` |

SPDX does not natively support BLAKE3. It is emitted under the `hashes` object with key `"blake3"`. This is a valid extension point per the SPDX specification.

#### 6.11.3 Download Location

For SPDX `downloadLocation`:
- `Source::Registry(url)` → `{url}{name}/-/{name}-{version}.tgz` (npm convention)
- `Source::Git(url)` → `git+{url}`
- `Source::Local(path)` → `file://{path}`
- `Source::Workspace` → `NOASSERTION`
- `Source::CasHttp(url)` → `{url}{name}/{version}`
- `Source::Ipfs(cid)` → `ipfs://{cid}`

If the registry URL ends with `/`, do not double-slash.

#### 6.11.4 Relationship / Dependency Mapping

For each package `P` with a dependency `D`:
- SPDX: `Relationship(P, DEPENDS_ON, D)`
- CycloneDX: `dependencies[P.purl].dependsOn.push(D.purl)`

Only `DepType::Runtime` and `DepType::Peer` dependencies are included in SBOM relationships. `Dev` dependencies are excluded by default. A future `SbomOptions` struct may allow including dev dependencies.

#### 6.11.5 Attestation Mapping

| HLOCK Attestation | SPDX | CycloneDX |
|---|---|---|
| `Attestation::None` | (omitted) | (omitted) |
| `Attestation::ExternalBundleSha256(hash)` | `originator: "NOASSERTION"`, `externalRefs` with `SECURITY` category pointing to bundle | `evidence.identity` with `field: "slsa"`, `confidence` |
| `Attestation::InlineSlsa(pred)` | `originator: "Organization: {builder}"`, `sourceInfo: "{source}"` | `evidence.identity` with `field: "slsa"`, `value: "{builder}"` |

#### 6.11.6 Package without Hashes

Packages with no `IntegrityHash` entries (e.g., workspace packages) are still included in the SBOM but with:
- SPDX: `filesAnalyzed: false`, no `packageVerificationCode`
- CycloneDX: no `hashes` array entry

### 6.12 Lint Engine Specification

#### 6.12.1 Built-in Rule Definitions

**`NoGitUrls`** (Severity: Error)
- For each package, check if `lockfile.sources[pkg.source_idx]` is `Source::Git`.
- Finding: `"Package '{name}' uses a git URL source which is non-reproducible"`

**`RequireIntegrity`** (Severity: Error)
- For each non-workspace package (`source != Source::Workspace`), check `pkg.hashes.len() > 0`.
- Finding: `"Package '{name}' has no integrity hashes"`

**`NoSha1`** (Severity: Warning)
- For each package, check if any `IntegrityHash` has `algo == HashAlgorithm::Sha1`.
- Finding: `"Package '{name}' uses deprecated SHA-1 hash"`

**`NoPeerAsRuntime`** (Severity: Warning)
- For each package, check if any dependency has `dep_type == DepType::Peer`.
- Finding: `"Package '{name}' has a peer dependency '{dep_name}' declared as a runtime dependency"`

**`MaxDepth`** (Severity: Warning)
- For each `ResolutionProvenance` entry, check `depth > max`.
- Finding: `"Package '{name}' is at depth {depth}, exceeding maximum {max}"`
- Configuration: `MaxDepth { max: u32 }`. Default recommended: `5`.

**`RequireAttestation`** (Severity: Info)
- For each non-workspace package with at least one `IntegrityHash`, check if any hash has `attestation != Attestation::None`.
- Finding: `"Package '{name}' has no supply chain attestation"`

#### 6.12.2 `lint()` Function

```rust
pub fn lint(lockfile: &Lockfile, rules: &[Box<dyn LintRule>]) -> LintReport
```

1. Create an empty `LintReport`.
2. For each rule in `rules`:
   - Call `rule.check(lockfile)`.
   - Extend `report.findings` with the results.
3. Sort findings by `(severity, package, rule)` for deterministic output.
4. Return the report.

#### 6.12.3 Convenience Function

```rust
pub fn lint_default(lockfile: &Lockfile) -> LintReport {
    let rules: Vec<Box<dyn LintRule>> = vec![
        Box::new(lint::NoGitUrls),
        Box::new(lint::RequireIntegrity),
        Box::new(lint::NoSha1),
        Box::new(lint::NoPeerAsRuntime),
        Box::new(lint::MaxDepth { max: 5 }),
        Box::new(lint::RequireAttestation),
    ];
    lint(lockfile, &rules)
}
```

#### 6.12.4 `LintReport` Methods

```rust
impl LintReport {
    pub fn has_errors(&self) -> bool {
        self.findings.iter().any(|f| f.severity == LintSeverity::Error)
    }

    pub fn errors(&self) -> impl Iterator<Item = &LintFinding> {
        self.findings.iter().filter(|f| f.severity == LintSeverity::Error)
    }

    pub fn warnings(&self) -> impl Iterator<Item = &LintFinding> {
        self.findings.iter().filter(|f| f.severity == LintSeverity::Warning)
    }

    pub fn is_empty(&self) -> bool {
        self.findings.is_empty()
    }
}
```

### 6.13 Provenance Method Implementations

```rust
impl Lockfile {
    pub fn provenance_for(&self, package_name: &str) -> Option<&ResolutionProvenance> {
        self.provenance.iter().find(|p| p.package_name == package_name)
    }

    pub fn packages_at_depth(&self, depth: u32) -> Vec<&ResolutionProvenance> {
        self.provenance.iter()
            .filter(|p| p.depth == depth)
            .collect()
    }

    pub fn dependency_chain(&self, package_name: &str) -> Vec<ResolutionProvenance> {
        let mut chain = Vec::new();
        let mut current = package_name;

        // Guard against cycles
        let mut visited = HashSet::new();

        while let Some(prov) = self.provenance.iter().find(|p| p.package_name == current) {
            if !visited.insert(current.to_string()) {
                break; // cycle detected
            }
            chain.push(prov.clone());
            if prov.constrained_by.is_empty() {
                break; // root dependency
            }
            current = &prov.constrained_by;
        }

        chain.reverse(); // root-first ordering
        chain
    }
}
```

**Invariants:**
- `provenance_for` returns `None` for packages without a `@provenance` directive.
- `dependency_chain` walks backward from the target to the root via `constrained_by`. It terminates on empty `constrained_by` (direct dependency), missing provenance, or cycle detection.
- The returned chain is ordered root-first: the first element is the direct dependency, the last is the target.

### 6.14 Serialization Order (Updated)

The `serialize` function emits lines in this order:

1. `@source` directives
2. `@override` directives
3. `@feature` directives
4. `@workspace-root` directive (if present)
5. `@workspace-pkg` directives
6. `@hoist-boundary` directives
7. `@metadata` directives
8. Empty line (separator)
9. Package lines (`name\t<payload>`, sorted by name)
10. `@artifact` directives
11. `@patch` directives
12. `@provenance` directives
13. `@digest` directive
14. `@signature` directives

The `@digest` is computed over bytes 1–12 (everything before `@digest`). The `@signature` signs bytes 1–13 (everything before `@signature`), consistent with v0.13 behavior.

---

### 6.15 Deserialization Updates

The `deserialize` function is updated to:

1. Parse `@provenance` lines encountered after `@patch` lines and before `@digest`/`@signature` lines.
2. Store parsed `ResolutionProvenance` entries in `lockfile.provenance`.
3. If duplicate `@provenance` directives exist for the same package name, the last one wins (per spec §3.1), and no error is raised. This is a deliberate design choice: merge tools may produce overlapping directives, and last-wins is deterministic.
4. Provenance entries referencing packages not in the lockfile are kept (not an error), but `lint::NoOrphanProvenance` may flag them.

**`@provenance` line parsing:**

```
@provenance <package> <constraint> <constrained_by> <dep_type_id> <source_type_id> <depth>
```

- `constraint` and `constrained_by` may be empty strings. An empty `constrained_by` means this is a direct (root) dependency.
- `dep_type_id` must be in `0..=4`. Otherwise `UnknownProvenanceDepType`.
- `source_type_id` must be in `0..=5`. Otherwise `UnknownProvenanceSourceType`.
- `depth` must be a valid decimal `u32`. Otherwise `InvalidHeader`.

## 8. Version Compatibility

| Aspect | v0.13 | v0.14 |
|---|---|---|
| Payload version byte | `0x08` | `0x08` (unchanged) |
| Binary payload layout | As specified in v0.12 | Unchanged |
| `@provenance` directive | Not recognized | Parsed on deserialize |
| `Lockfile.provenance` | N/A | `Vec<ResolutionProvenance>` |
| `@digest` boundary | Before `@digest` | Before `@digest` (unchanged) |
| `@signature` signs over | Everything before `@signature` | Unchanged |
| v0.13 lockfile deserialized by v0.14 | ✅ | `provenance` field is `vec![]` |
| v0.14 lockfile deserialized by v0.13 | ✅ | `@provenance` lines skipped (unknown lines after packages are not an error in v0.13's parser... **wait, they would be**). |

**Forward compatibility concern:** v0.13's `deserialize` will encounter `@provenance` lines and treat them as errors (falling through to the `else` branch in `parse_header` or being rejected in the package parsing loop). This is acceptable per the project's versioned-release model: v0.14 consumers must use v0.14 libraries. However, the `@provenance` lines are placed *after* `@patch` lines and *before* `@digest`/`@signature`, in the same region where `@artifact` and `@patch` already live. Reviewing the v0.13 `deserialize` code, lines after the package section that start with `@artifact` and `@patch` are handled. Lines starting with `@signature` and `@digest` are skipped. Any other `@`-prefixed line in that region would fall through and be treated as a package line (missing tab delimiter → `MissingDelimiter` error).

**Mitigation:** Update v0.13's `deserialize` to skip unknown `@`-prefixed directives in the package section, OR accept that v0.13 cannot read v0.14 files. Given the semantic versioning convention (0.x = pre-release, no compatibility guarantee), this is acceptable. The spec notes this explicitly:

> **v0.13 cannot deserialize v0.14 lockfiles that contain `@provenance` directives.** This is intentional. Downgrade by stripping `@provenance` lines if needed.

## 10. Implementation Order

The recommended implementation order, respecting dependency chains:

1. **`provenance.rs`** — Types (`ResolutionProvenance`, `ProvenanceSourceType`) and `Lockfile` methods. No I/O dependencies.
2. **`error.rs`** — Add new error variants. No logic dependencies.
3. **`lockfile.rs`** — Update `serialize`/`deserialize` to handle `@provenance`. Update `Lockfile` struct. Update `extract_subgraph` / `extract_subgraph_platform` to filter provenance.
4. **`merge.rs`** — Depends on updated `Lockfile`.
5. **`lazy.rs`** — Depends on `lockfile.rs` header parsing and `payload.rs` for `unpack_payload`.
6. **`sbom.rs`** — Depends on `Lockfile`.
7. **`lint.rs`** — Depends on `Lockfile` and `provenance.rs`.
8. **`lib.rs`** — Re-exports.
9. **Integration tests** — End-to-end coverage.
10. **Unit tests** — Per-module coverage.
